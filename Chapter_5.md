## Конспект главы 5: Системы управления базами данных (СУБД)

### Определение СУБД

**СУБД** (система управления базами данных) — совокупность языковых и программных средств, обеспечивающих:

1. **Создание** БД.
2. **Поддержание (редактирование)** БД.
3. **Доступ** к данным со стороны пользователей и приложений.

### Что предоставляет СУБД

| Компонент | Что даёт |
|---|---|
| Средства поддержки логической модели данных | Возможность описывать структуру данных (таблицы, связи, ключи) |
| Развитый пользовательский интерфейс | Формы, меню, отчёты, графические средства |
| Средства программирования высокого уровня | Языки (SQL, расширения), возможность создавать собственные приложения |
| Средства администрирования, обеспечения целостности, безопасности и секретности | Управление пользователями, резервное копирование, аудит |

---

## 5.1. Архитектура построения СУБД

### Классы операций (функций) СУБД

| Класс | Операции |
|---|---|
| **a) Определение структуры данных** | Создание, изменение, удаление таблиц, индексов, связей (DDL) |
| **b) Изменение данных (обновление)** | Добавление (INSERT), удаление (DELETE), модификация (UPDATE) |
| **c) Поиск и доступ к информации** | Выборка (SELECT), поиск (установка курсора по критерию), сортировка, фильтрация |
| **d) Поддержка целостности и восстановления** | Транзакции, журналирование, контрольные точки |
| **e) Контроль доступа** | Аутентификация, авторизация, разграничение прав |
| **f) Поддержка обмена данными** | Импорт, экспорт, репликация |
| **g) Администрирование** | Управление пользователями, резервное копирование, настройка |

### Низкоуровневые операции (внутренние)

- Управление данными во внешней памяти.
- Управление буферами оперативной памяти.
- Управление параллельностью и транзакциями.
- Ведение журнала транзакций.

---

### Языки СУБД

| Язык | Расшифровка | Назначение | Тип |
|---|---|---|---|
| **DDL** | Data Definition Language (язык описания данных) | Описание логической структуры данных (создание таблиц, индексов, связей) | Декларативный (непроцедурный) |
| **DML** | Data Manipulation Language (язык манипулирования данными) | Выполнение операций с данными: ввод, удаление, модификация, выборка | Декларативный (указывает *что*, а не *как*) |

---

### Требования к СУБД (обязательные)

1. Возможность создавать новые БД и определять их схему; поддерживать разнообразные представления одних и тех же данных.
2. Позволять «запрашивать» данные и изменять их через DML; допускать интеграцию и совместное использование данных разными приложениями.
3. Поддерживать хранение очень больших массивов данных в течение долгого времени, гарантируя безопасность и целостность.
4. Контролировать конкурентный многопользовательский доступ к данным (управление параллельным доступом).

### Структурная схема СУБД (Рис. 23)

```
┌─────────────────────────────────────────────────────────────┐
│                      ПОЛЬЗОВАТЕЛИ                            │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                   ПОЛЬЗОВАТЕЛЬСКИЙ ИНТЕРФЕЙС                 │
│              (формы, меню, отчёты, графические средства)      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│              ИНТЕРПРЕТАТОР DDL / ПРЕДКОМПИЛЯТОР DML           │
│    (обработка инструкций определения и манипулирования)       │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                        ЯДРО (ПРОЦЕССОР) БД                   │
│    (управление данными, буферами, транзакциями, индексами)    │
└─────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┼───────────────┐
              ▼               ▼               ▼
┌─────────────────┐ ┌─────────────────┐ ┌─────────────────┐
│  СЛОВАРЬ ДАННЫХ │ │  БАЗА ДАННЫХ    │ │  ЖУРНАЛ         │
│  (метаданные)   │ │  (пользователь- │ │  (транзакций)   │
│                 │ │   ские данные)  │ │                 │
└─────────────────┘ └─────────────────┘ └─────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│                    ОПЕРАЦИОННАЯ СИСТЕМА                       │
│               (управление файлами и вводом-выводом)           │
└─────────────────────────────────────────────────────────────┘
```

**Пояснение блоков:**

| Блок | Назначение |
|---|---|
| **Пользовательский интерфейс** | Формы, меню, отчёты — то, с чем работает пользователь |
| **Интерпретатор DDL** | Обрабатывает команды создания/изменения структуры, сохраняет результаты в словарь данных |
| **Предкомпилятор DML** | Преобразует абстрактные запросы пользователя в инструкции DML |
| **Ядро (процессор) БД** | Главный движок: управляет данными, буферами, транзакциями, индексами, целостностью, безопасностью |
| **Словарь данных (системный каталог)** | Хранит **метаданные** — описание структуры БД (таблицы, поля, типы, связи, индексы) |
| **База данных** | Хранит пользовательские данные на внешних носителях |
| **Журнал транзакций** | Записывает все изменения для восстановления после сбоев |
| **Операционная система** | Обеспечивает низкоуровневый доступ к файлам (чтение/запись). При необходимости СУБД может управлять дисковыми операциями сама (для производительности) |
| **Буфер (ОП)** | Область в оперативной памяти для ускорения доступа. Скорость ОП >> скорость внешней памяти, буфер позволяет кэшировать часто используемые данные |

---

### Классификация СУБД (признаки различия)

| Признак | Виды |
|---|---|
| **По типу поддерживаемой модели данных** | Реляционная, объектная, объектно-реляционная, иерархическая, сетевая, многомерная, постреляционная |
| **По количеству поддерживаемых пользователей** | Однопользовательские, многопользовательские |
| **По количеству компьютеров** | Централизованные (один компьютер), распределённые (DDBMS) |
| **По назначению** | Общецелевые, специальные (банки, билетные системы, OLTP-системы) |

**OLTP** — On-Line Transaction Processing системы (оперативная обработка транзакций).

## 5.2. Индексы

### Назначение индексов

1. **Ускорение доступа к данным и повышение скорости поиска** — устранение последовательного (пошагового) сканирования записей.
2. **Проверка уникальности** значений поля.
3. **Сортировка записей** в таблице (без физического перемещения записей).

> **Важно:** индекс ≠ первичный ключ. Индекс — гораздо более сложный объект.

### Принцип работы индекса

Индекс можно представить как таблицу из двух столбцов:

| Столбец 1 | Столбец 2 |
|---|---|
| Упорядоченные значения индексного выражения | Местоположение записи (указатель на блок данных) |

**Реальная реализация:** чаще всего — сбалансированное двоичное дерево (**B-tree**).

**Скорость поиска:** при использовании бинарного дерева число обращений к индексу = **log₂ n**, где n — число записей в индексе.

### Хеширование как альтернатива

Вместо хранения самих значений можно хранить **хеш-код** (свёртку) — число фиксированной длины (например, 4 байта).

| Преимущества | Недостатки |
|---|---|
| Длина хеш-кода не зависит от длины исходного значения → экономия места и времени поиска | Необходимость вычислять хеш-код (затраты времени) |
| — | Возможность **коллизий** (разные значения → одинаковый хеш-код) |

---

### 5.2.1. Первичные индексы

**Определение:** индекс, созданный на основе **первичного ключа** таблицы. Поле индекса определяет **физический порядок** записей в таблице.

**Ограничение:** для одной таблицы может быть определён **только один** первичный индекс.

**Структура первичного индекса (Рис. 24):**

```
┌──────────────────────────────────────────────────────────────┐
│                     ПЕРВИЧНЫЙ ИНДЕКС                         │
├─────────────────────┬────────────────────────────────────────┤
│ Первое поле         │ Второе поле                            │
│ (значение индекса)   │ (указатель на блок данных)            │
├─────────────────────┼────────────────────────────────────────┤
│ Иванов              │ → Блок 1                               │
│ Петров              │ → Блок 2                               │
│ Сидоров             │ → Блок 3                               │
└─────────────────────┴────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│                   БЛОКИ ДАННЫХ (физический носитель)          │
├─────────────────────┬─────────────────────┬─────────────────┤
│ Блок 1              │ Блок 2              │ Блок 3          │
│ (Иванов, ...)       │ (Петров, ...)       │ (Сидоров, ...)  │
└─────────────────────┴─────────────────────┴─────────────────┘
```

**Важная особенность:** в первичном индексе хранится **не каждое** значение индексируемого поля, а только **первое значение для каждого блока данных**. Число записей в первичном индексе = числу блоков данных.

**Алгоритм поиска:**
1. Поиск в индексе (бинарный) → находим нужный блок.
2. Последовательный поиск внутри блока (блок загружается в буфер ОП).

---

### 5.2.2. Вторичные индексы

**Определение:** индекс, основанный на **неупорядоченных физически** полях (не на первичном ключе).

**Отличия от первичного индекса:**

| Аспект | Первичный индекс | Вторичный индекс |
|---|---|---|
| Физический порядок записей | Определяет | Не определяет |
| Количество на таблицу | Только один | Несколько |
| Что хранится во втором поле | Указатель на блок | Указатели на **все** блоки с данным значением |
| Повторяющиеся значения | Нет (первичный ключ уникален) | Есть → список указателей |

**Структура вторичного индекса (Рис. 25):**

```
┌──────────────────────────────────────────────────────────────┐
│                    ВТОРИЧНЫЙ ИНДЕКС                          │
├─────────────────────┬────────────────────────────────────────┤
│ Значение индекса    │ Список указателей на блоки             │
├─────────────────────┼────────────────────────────────────────┤
│ Минск               │ → Блок 1, Блок 3, Блок 7               │
│ Гродно              │ → Блок 2                               │
│ Брест               │ → Блок 4, Блок 5                       │
└─────────────────────┴────────────────────────────────────────┘
```

---

### 5.2.3. Составные индексы

**Когда применяются:** когда поиск часто ведётся по **нескольким полям одновременно**.

**Отличие от обычного индекса:** первое поле индекса содержит **свёртку значений всех полей**, входящих в индекс.

**Порядок сортировки:**
1. Сортировка по первому полю (например, по Страна).
2. Внутри — по второму полю (Город).
3. Внутри — по третьему (Предприятие).

**Пример:** поиск по условию:
```
Страна = 'Беларусь' AND Город = 'Минск' AND Предприятие < 'С'
```

**Важное ограничение:** в большинстве СУБД **только последнее условие** в составном индексе может быть неравенством (<, >, ≤, ≥). Все предыдущие — только равенство (=).

**Почему несколько вторичных индексов не заменят один составной:**
- Результат **вложенной сортировки** (сначала по одному полю, потом по другому) ≠ результату **последовательной сортировки** (одновременно по всем полям).

---

### 5.2.4. Хеш-индексирование

**Определение:** технология быстрого прямого доступа к записи, при которой адрес записи вычисляется как **хеш-функция от значения поля** (хеш-поля).

**Алгоритм:**
1. **Сохранение записи:** вычислить хеш-адрес → поместить запись по этому адресу.
2. **Извлечение записи:** вычислить хеш-адрес по заданному значению → прочитать запись по этому адресу.

**Требования к хеш-функции h(K):**
1. Обеспечивать **равномерное распределение** записей в памяти (близкие значения ключа → сильно отличающиеся адреса).
2. Для разных значений ключа выдавать **разные адреса** (на практике допускаются коллизии).

**Пример хеш-функции:** `h(K) = K mod M`, где M — количество записей.

**Достоинства хеш-индексирования:**
- Обращение к данным по значению ключа — **за одну операцию ввода/вывода**.
- Не нужно создавать дополнительные структуры (индексы) и тратить память.

**Недостатки хеш-индексирования:**
1. Количество записей и распределение значений хеш-поля должны быть **известны заранее**.
2. Записи **неупорядочены** по хеш-полю → дополнительные затраты при сортировке.
3. Сложность подбора хеш-функции (количество значений может значительно превышать число адресуемых страниц).
4. **Коллизии** (совпадение адресов для разных записей). С ростом числа записей коллизий становится больше → время поиска растёт.

### Вопрос 10. Для чего используются многоуровневые индексы?

*(В данной главе этот вопрос прямо не раскрыт, но по смыслу — для ускорения поиска в очень больших индексах, когда один уровень индекса не помещается в ОП. Создаётся индекс на индекс — многоуровневое дерево).*

### Вопрос 11. Поясните, почему скорость поиска с использованием первичного индекса является наибольшей.

1. Первичный индекс имеет **наименьший размер** (количество записей = количество блоков данных, а не записей).
2. Поиск в индексе — **бинарный** (log₂ n обращений).
3. Найденный блок загружается в буфер, внутри блока — последовательный поиск (блок мал).
4. Записи физически упорядочены по ключу → минимум перемещений головки диска.

### Вопрос 12. Чем отличается первичный индекс от вторичного?

| Аспект | Первичный индекс | Вторичный индекс |
|---|---|---|
| Физический порядок | Определяет | Не определяет |
| Количество на таблицу | Один | Несколько |
| Уникальность значений | Да (первичный ключ) | Могут быть повторы |
| Второе поле индекса | Указатель на **блок** | **Список** указателей на блоки |
| Размер | Меньше (число блоков) | Больше (может быть как число записей) |

### Вопрос 13. Чем отличается составной индекс от простого?

| Аспект | Простой индекс | Составной индекс |
|---|---|---|
| Число полей | Одно | Несколько |
| Сортировка | По одному полю | Многоуровневая (по первому, затем по второму и т.д.) |
| Первое поле индекса | Значение поля | Свёртка значений **всех** полей |
| Применение | Поиск по одному полю | Поиск по комбинации полей |
| Ограничение | — | Только последнее условие может быть неравенством |

### Вопрос 14. Каким образом происходит ускорение поиска записей при хеш-индексировании?

**Механизм:** адрес записи вычисляется непосредственно из значения ключа с помощью хеш-функции, а не находится через поиск в индексе.

1. При сохранении: `адрес = h(значение_ключа)` → запись кладётся по этому адресу.
2. При поиске: по заданному значению ключа вычисляется тот же адрес → запись читается за **одну операцию ввода/вывода**.

**Скорость:** обращение к данным по ключу происходит за **O(1)** в идеальном случае (без коллизий).

## Дополнительные важные термины (для зачёта)

| Термин | Определение |
|---|---|
| **DDL** | Язык описания данных (создание, изменение структуры БД) |
| **DML** | Язык манипулирования данными (INSERT, DELETE, UPDATE, SELECT) |
| **Словарь данных (системный каталог)** | Хранилище метаданных — описания структуры БД |
| **Ядро (процессор) БД** | Центральный компонент СУБД, управляющий данными, транзакциями, буферами |
| **Буфер** | Область в ОП для кэширования данных с диска |
| **Первичный индекс** | Индекс по первичному ключу, определяющий физический порядок записей |
| **Вторичный индекс** | Индекс по неключевому полю, не влияющий на физический порядок |
| **Составной индекс** | Индекс по нескольким полям для поиска по комбинации |
| **B-tree** | Сбалансированное двоичное дерево — структура данных для индексов |
| **Хеширование** | Преобразование данных произвольной длины в число фиксированной длины |
| **Коллизия** | Ситуация, когда разным значениям соответствует одинаковый хеш-адрес |
| **OLTP** | On-Line Transaction Processing — системы оперативной обработки транзакций |
