## Вопрос 1. Понятия базы данных (БД) и системы управления базами данных (СУБД)

### 📚 Подробное раскрытие по материалу пособия

**Определение базы данных (БД):**

Согласно пособию, в общем смысле термин «база данных» можно применить к любой совокупности связанных данных, объединённых по определённому признаку и предназначенных для решения некоторой задачи (стр. 3).

Более строгое классическое определение дано Дж. Мартином (стр. 3):  
> *«БД – это совокупность взаимосвязанных данных при такой их минимальной избыточности, которая допускает их использование оптимальным образом для одного или нескольких приложений в определённой предметной области человеческой деятельности»*.

Также на стр. 3 уточняется, что БД — это динамическая информационная модель некоторой предметной области деятельности человека. Под предметной областью понимается часть реального мира, представляющая интерес для исследования.

Важное замечание (стр. 4): не всякий набор данных можно назвать БД. Данные должны быть логически упорядочены и структурированы.

**Определение СУБД:**

На стр. 4 пособие даёт следующее определение:  
*«Под СУБД понимают совокупность языковых и программных средств, обеспечивающих создание, поддержание и доступ к данным как со стороны пользователей, так и со стороны приложений»*.

Кроме того, на стр. 4 указано, что СУБД предоставляет:
- развитый пользовательский интерфейс;
- средства программирования высокого уровня;
- средства администрирования;
- обеспечение секретности и безопасности информации.

**Различие БД и СУБД (стр. 4):**  
- **БД** — это набор взаимосвязанных данных, хранящихся на физических носителях.
- **СУБД** — это программная среда, позволяющая управлять многими БД и имеющая средства для их разработки.

На Рис. 1 (стр. 4) показана структура БД, включающая:
- ядро (процессор) БД;
- набор данных;
- программу интерфейса с пользователем.

### ✅ Итог простыми словами

> **База данных (БД)** — это просто хранилище связанных между собой данных (например, список студентов с их оценками).  
> **Система управления базами данных (СУБД)** — это программа, которая помогает создавать БД, искать в них информацию, добавлять, изменять или удалять данные, а также защищать их от посторонних.  
> Проще говоря: БД — это «склад» данных, а СУБД — «управляющий» на этом складе.

## Вопрос 2. Основные свойства баз данных

### 📚 Подробное раскрытие по материалу пособия

На стр. 4–5 пособие перечисляет основные требования к набору данных, чтобы его можно было назвать БД. Эти требования и определяют свойства технологии баз данных:

1. **Связность, логическая упорядоченность и структурированность** (стр. 4)  
   Беспорядочный набор данных не может считаться БД. Например, слова в документе — не БД. Данные должны быть однородными (в поле «Фамилия» — только фамилия).

2. **Контролируемая избыточность данных (Controlling Redundancy)** (стр. 4)  
   Каждый объект должен быть описан только один раз. Некоторая избыточность допускается только для повышения скорости работы и восстановления после сбоев, но она должна контролироваться.

3. **Целостность (Data Integrity)** (стр. 5)  
   Подразумевает правильность данных в любой момент времени.

4. **Физическая и логическая независимость** (стр. 5)  
   - *Физическая независимость*: данные не зависят от того, где и как они хранятся на диске.  
   - *Логическая независимость*: приложения не зависят от изменений в логической структуре данных (можно добавлять/удалять поля, не переделывая всю БД).

5. **Безопасность и секретность** (стр. 5)  
   - *Безопасность*: защита данных от случайного или преднамеренного доступа, модификации или разрушения.  
   - *Секретность*: право решать, кому, когда и сколько информации можно передать.

Дополнительно на стр. 14 (глава 1, итог) перечислены **шесть основных характеристик** БД как технологии хранения:
1. Структурированное хранение данных при их минимальной избыточности.
2. Использование метаданных для описания БД.
3. Независимость программ и данных (и независимость программ и операций).
4. Наличие встроенных средств обеспечения целостности, безопасности и секретности.
5. Поддержка разнообразных способов отображения и выборки данных.
6. Наличие расширенных средств поиска информации.

### ✅ Итог простыми словами

> У настоящей базы данных есть 5 главных свойств:  
> 1. **Данные упорядочены** — не просто куча информации, а всё разложено по полочкам.  
> 2. **Нет лишних повторов** — каждая запись хранится один раз.  
> 3. **Данные правильные** — система следит, чтобы ошибок не было.  
> 4. **Независимость** — можно менять, где и как хранятся данные, не ломая программы.  
> 5. **Безопасность** — доступ только у тех, у кого есть разрешение.

## Вопрос 3. Трёхуровневая архитектура построения баз данных (ANSI/SPARC)

### 📚 Подробное раскрытие по материалу пособия

На стр. 8–9 и Рис. 2 описана трехуровневая архитектура, предложенная в 1978 г. как стандарт ANSI/SPARC. Она поддерживается большинством коммерческих СУБД.

**Уровни архитектуры:**

1. **Внутренний уровень (internal)** (стр. 8)  
   Определяется физической моделью данных. Описывает:
   - размещение данных на физических носителях;
   - способы доступа к данным;
   - структуру файлов, индексов.

2. **Концептуальный уровень (conceptual)** (стр. 8)  
   Определяется концептуальной моделью данных. Описывает логическую структуру данных **без указания деталей физического хранения**.

3. **Внешний уровень (внешний, или уровень представлений/интерфейса)** (стр. 9)  
   Выводит необходимые данные в требуемом формате, скрывая остальную часть БД.  
   - Каждый пользователь видит только то, что ему нужно.  
   - Можно менять своё представление, не влияя на других.  
   - Разные пользователи могут использовать разные языки общения с БД (меню, SQL, расширения SQL и т.д.).

**Связь между уровнями** (стр. 9):  
Программы отображения транслируют запросы с одного уровня на другой. Запрос пользователя → интерпретация на концептуальном уровне → преобразование в физический запрос → извлечение данных → преобразование к нужному виду.

Рис. 2 (стр. 9) наглядно показывает три уровня и связь:  
- Внутренний уровень — «сохранённые данные»;  
- Концептуальный уровень — «концептуальная схема данных»;  
- Внешний уровень — «внешнее представление» для разных пользователей.

**Независимость данных** (стр. 10):  
- *Логическая независимость* обеспечивается развязкой внешнего представления от способа организации данных.  
- *Физическая независимость* обеспечивается развязкой внутреннего и концептуального уровней.

### ✅ Итог простыми словами

> Трёхуровневая архитектура — это принцип организации БД, похожий на матрёшку:  
> - **Внутренний уровень** — как реальное расположение вещей на складе (где что лежит).  
> - **Концептуальный уровень** — общая схема склада без деталей (что вообще есть).  
> - **Внешний уровень** — что видит конкретный человек (менеджер видит товары, бухгалтер — цены).  
> Благодаря этому можно менять внутреннее устройство, не меняя то, как видят данные пользователи.

## Вопрос 4. Понятие целостности данных

### 📚 Подробное раскрытие по материалу пособия

На стр. 11 (раздел 1.3 «Обеспечение целостности и правильности данных») даётся следующее определение:

> *«Целостность есть свойство БД, означающее, что в ней содержится полная, непротиворечивая, согласованная и адекватно отражающая предметную область информация»*.

Иными словами, под целостностью данных понимают соответствие информационной модели предметной области, то есть данных, хранимых в БД, объектам реального мира и их взаимосвязям в каждый момент времени (стр. 11).

**Важные положения из пособия (стр. 11):**
- Любое изменение в предметной области, значимое для модели, должно отражаться в БД.
- При этом должна сохраняться однозначная интерпретация информационной модели в терминах предметной области.
- В пособии приводится аналогия с книгоизданием: данные в книге проверяют несколько редакторов. Люди склонны слепо верить компьютеру, но если в компьютер заложены неверные данные или произошёл сбой, то о достоверности можно забыть.

**Поддержание целостности включает** (стр. 12):
- проверку целостности;
- восстановление целостности в случае обнаружения противоречий.

### ✅ Итог простыми словами

> **Целостность данных** — это свойство базы данных, означающее, что все данные в ней правильные, непротиворечивые и соответствуют реальному миру. Например, если у человека в БД указан день рождения 31 июня — это нарушение целостности, потому что такого дня не бывает. Система следит, чтобы таких ошибок не возникало.

## Вопрос 5. Виды ограничителей целостности

### 📚 Подробное раскрытие по материалу пособия

На стр. 12 пособие перечисляет **три типа ограничителей целостности** (constraints):

1. **Ограничители значений**  
   К ним относятся:
   - задание типа данных (например, число, текст, дата);
   - маска ввода, позволяющая ввод только определённых данных (например, шаблон для телефона);
   - задание диапазона значений (например, возраст от 0 до 120);
   - задание списка значений (например, пол: только «М» или «Ж»).

2. **Ссылочная целостность**  
   Обеспечивается контролем отношений между связанными данными. Включает:
   - контроль связей между таблицами;
   - введение каскадного удаления и обновления связанных записей (например, если удалить заказ, то удалятся и связанные с ним позиции).

3. **Целостность записи**  
   В минимальном объёме требований заключается в требовании существования записи. Обеспечивается:
   - объявлением обязательных к вводу данных;
   - выделением семантически важных полей (например, нельзя добавить студента без указания фамилии).

### ✅ Итог простыми словами

> **Ограничители целостности** — это правила, которые следят за правильностью данных.  
> - **Ограничители значений**: проверяют, что число не выходит за пределы, а дата — реальна.  
> - **Ссылочная целостность**: следят, чтобы ссылки между таблицами не «обрывались» (например, нет заказа без клиента).  
> - **Целостность записи**: требуют заполнить обязательные поля.

## Вопрос 6. Понятие транзакции

### 📚 Подробное раскрытие по материалу пособия

На стр. 12 даётся следующее определение:

> *«Транзакцией называется некоторая неделимая последовательность операций над данными БД, которая отслеживается от начала и до завершения»*.

**Ключевая особенность** (стр. 12):  
Если по каким-либо причинам (сбои, ошибки) транзакция остаётся незавершённой, то производится **отмена всех операций**, входящих в её состав. То есть транзакция либо выполняется полностью, либо не выполняется вообще («всё или ничего»).

**Важность в многопользовательских БД** (стр. 12–13):  
Контроль транзакций особенно важен, когда несколько пользователей одновременно работают с одними и теми же данными. Если транзакции запущены параллельно, порядок выполнения их частей может играть решающую роль — от него зависит конечное состояние данных.

**Сериализация транзакций** (стр. 13):  
Под сериализацией понимают составление такого плана выполнения транзакций (последовательного), при котором суммарный эффект эквивалентен эффекту их поочередного выполнения. При параллельном выполнении возможны конфликты (блокировки), тогда одна или несколько транзакций откатываются.

### ✅ Итог простыми словами

> **Транзакция** — это группа операций с БД, которая выполняется как одно целое. Если хоть одна операция не удалась (например, сбой электричества), то не выполняются и все остальные — база данных возвращается в то состояние, которое было до начала транзакции. Это как перевод денег: если деньги списались с одного счёта, но не зачислились на другой, вся операция отменяется.

## Вопрос 7. Свойства транзакции (ACID)

### 📚 Подробное раскрытие по материалу пособия

На стр. 12 пособие перечисляет свойства, известные как требования **ACID**:

1. **Атомарность (atomicity)**  
   Выполняются все входящие в транзакцию операции или ни одна. Транзакция неделима.

2. **Согласованность (consistency)**  
   Любая транзакция должна переводить БД из одного согласованного состояния в другое согласованное состояние. Нарушение ограничений целостности не допускается.

3. **Изолированность (isolation)**  
   Транзакции выполняются независимо друг от друга. Внешне параллельное выполнение даёт тот же результат, что и последовательное. В случае конфликтов производится откат.

4. **Безопасность (durability — долговечность)**  
   Даже аварийное завершение работы (сбой питания, ошибка ОС) не приводит к потере данных. Результаты завершённой транзакции сохраняются навсегда.

На стр. 13 дополнительно поясняется, что в многопользовательских БД при параллельном доступе к одним и тем же данным порядок выполнения частей транзакций важен для конечного состояния данных. Для этого используется сериализация и механизмы блокировок.

### ✅ Итог простыми словами

> **ACID** — это четыре главных свойства любой транзакции:  
> - **Атомарность**: всё или ничего (никаких «половинчатых» изменений).  
> - **Согласованность**: транзакция не нарушает правила целостности БД.  
> - **Изолированность**: транзакции не мешают друг другу, даже если выполняются одновременно.  
> - **Долговечность**: если транзакция завершилась успешно, результат сохранится даже при выключении электричества.

## Вопрос 8. Понятие модели данных

### 📚 Подробное раскрытие по материалу пособия

На стр. 15 (раздел 2.1) даётся следующее определение:

> *«Модель данных – это средство абстракции (иначе формальная теория), позволяющее отобразить информационное содержание данных»*.

Важное уточнение (стр. 15): результат моделирования данных также часто называют моделью данных, который может быть представлен простым перечнем информации, подлежащей хранению в информационной системе. Однако, поскольку данные всегда структурированы и тесно взаимосвязаны, необходимо отражать и их структуру.

**Ключевые положения (стр. 15):**
- Модель данных представляет собой **совокупность правил**, которые могут быть использованы для:
  - описания структуры данных;
  - контроля целостности данных.
- Как теория, модель данных должна иметь средства для:
  - описания логической структуры данных;
  - правил контроля целостности;
  - операций определения и манипулирования данными.

**Важное замечание (стр. 15):**  
Моделей для отображения одной и той же информации можно придумать множество. **Критерием выбора** будет максимум количества информации, извлекаемой из данных.

### ✅ Итог простыми словами

> **Модель данных** — это способ описать, какие данные мы храним, как они связаны между собой и какие правила должны соблюдаться (например, чтобы данные были правильными). Это как чертёж или схема, по которой строится база данных. Если данных много и они запутанные, модель помогает их упорядочить.

## Вопрос 9. Виды атрибутов

### 📚 Подробное раскрытие по материалу пособия

На стр. 16–17 (раздел 2.1) подробно описаны виды атрибутов.

**Определение атрибута (стр. 16):**  
*«Атрибут – это информационное отражение свойств объекта»*. Каждый объект характеризуется рядом основных атрибутов (например, для человека: ФИО, год рождения, номер паспорта, адрес).

**Виды атрибутов:**

1. **Простые и составные (стр. 16)**  
   - *Простые*: неделимые, например, «год рождения».  
   - *Составные*: складываются из нескольких простых, например, «адрес» (улица, дом, квартира).  
   - Рекомендация: если возможны ссылки на часть составного атрибута, лучше разложить его на простые.

2. **Однозначные и многозначные (стр. 16–17)**  
   - *Однозначные*: принимают только одно значение.  
   - *Многозначные*: могут принимать набор значений. Например, «телефоны клиента» или «адреса клиента».

3. **Сохраняемые и вычисляемые (производные) (стр. 17)**  
   - *Сохраняемые*: хранятся непосредственно в БД.  
   - *Вычисляемые*: могут быть вычислены на основе уже существующих атрибутов. Например, возраст человека вычисляется из текущей даты и дня рождения.

4. **Пустое значение (Null) (стр. 17)**  
   Специальное пустое значение вводится для атрибутов, которые не имеют значений или могут быть опущены при вводе. Различают три случая:
   - мы не знаем значение атрибута;
   - атрибут не может иметь значения в данном конкретном случае (например, учёная степень для сотрудника без степени);
   - значение известно, но пропущено при вводе.

**Домен (стр. 17):**  
Домен — это набор всех потенциальных значений, которые могут быть присвоены атрибуту. Например, домен «Имена» содержит все возможные имена. Домен — это, по сути, пользовательский тип данных (как перечисление enum в C).

**Ключевой атрибут (стр. 17–18):**  
Атрибут, по которому можно однозначно идентифицировать остальные атрибуты объекта, называется **ключевым атрибутом**. Он может быть составным (из нескольких атрибутов). Если таких атрибутов несколько, один выбирают первичным ключом, остальные — альтернативными (потенциальными) ключами.

**Запись данных (стр. 18):**  
Совокупность значений атрибутов объекта называется **записью данных** (record).

### ✅ Итог простыми словами

> **Атрибуты** — это свойства объектов (например, у студента: имя, возраст, группа).  
> Они бывают:
> - **простые** (одно поле) и **составные** (из нескольких частей);
> - **однозначные** (одно значение) и **многозначные** (несколько телефонов);
> - **сохраняемые** (хранятся в БД) и **вычисляемые** (например, возраст);
> - могут быть **пустыми (Null)** — когда значение неизвестно или не применимо.  
> **Ключевой атрибут** — это уникальный «идентификатор» объекта (например, паспорт или номер студенческого).

## Вопрос 10. Функциональные, транзитивные и многозначные зависимости атрибутов

### 📚 Подробное раскрытие по материалу пособия

На стр. 19–21 (раздел 2.2) подробно описаны виды зависимостей между атрибутами.

**1. Функциональная зависимость (стр. 19–20)**  

Определение: *«Атрибут В функционально зависит от атрибута А, если каждому значению атрибута А соответствует одно и только одно значение атрибута В»*. Обозначение: **А → В**.

Пример: знание номера паспорта (А) однозначно определяет фамилию человека (В).  
Обратное неверно: одной фамилии может соответствовать много разных паспортов.

Особые случаи:
- **Функциональная взаимозависимость (А ↔ В)** — если А → В и В → А (взаимно-однозначное соответствие). Приводит к связи «один к одному» между атрибутами.
- **Частичная функциональная зависимость** — зависимость от части составного атрибута.
- **Полная функциональная зависимость** — зависимость от всего составного атрибута целиком.

**2. Транзитивная зависимость (стр. 20)**  

Определение: *«Атрибут С зависит от атрибута А транзитивно (А →→ С), если для атрибутов А, В, С выполняются условия А → В и В → С, но обратная зависимость отсутствует»*.

Пример: ФИО → Должность → Оклад. Зная ФИО, можно определить должность, а по должности — оклад. Но оклад не напрямую зависит от ФИО (в отличие от должности).

**3. Многозначная зависимость (стр. 20–21)**  

Определение: *«Атрибут В многозначно зависит от атрибута А, если каждому значению А соответствует множество значений В»*.

Обозначения:
- А «– В — связь «один ко многим»;
- А –» В — связь «многие к одному»;
- А «–» В — связь «многие ко многим».

Пример: номер группы «–» фамилия студента (в одной группе много студентов).  
Пример: студент и предметы — многие ко многим (один студент сдаёт много предметов, один предмет сдают многие студенты).

**Тривиальные и нетривиальные многозначные зависимости (стр. 21):**
- **Тривиальная**: Y ⊂ X или Y ∪ X = R (все атрибуты множества R). Такие зависимости всегда выполняются.
- **Нетривиальная**: не выполняется ни одно из условий.

**Независимые атрибуты (стр. 21):**  
Два или более атрибута взаимно независимы, если ни один из них не является функционально зависимым от другого.

**Практический вывод (стр. 21–22):**  
При проектировании концептуальной модели необходимо выявить зависимости и перестраивать объекты до тех пор, пока все неключевые атрибуты не будут характеризоваться только полной функциональной зависимостью от соответствующих ключевых атрибутов (детерминантов). Тривиальные зависимости можно исключить из анализа.

### ✅ Итог простыми словами

> **Зависимости между атрибутами** — это правила, как одни данные связаны с другими.
> - **Функциональная зависимость** (А → В): по значению А всегда можно определить В. Например, по номеру паспорта → фамилию.
> - **Транзитивная зависимость** (А →→ С): А определяет В, В определяет С, но А напрямую С не определяет. Например, должность влияет на оклад, но оклад не напрямую зависит от имени.
> - **Многозначная зависимость**: одному значению А соответствует много значений В. Например, одна группа → много студентов.

## Вопрос 11. Типы связей между объектами

### 📚 Подробное раскрытие по материалу пособия

На стр. 18–19 (раздел 2.2) описаны типы связей.

**Общая классификация по количеству связываемых объектов (стр. 18):**  
Связи бывают **унарные** (один объект, иерархия подчинённости), **бинарные** (два объекта), **тернарные** (три) и более высоких порядков. Наибольший интерес представляют **бинарные связи**, так как связи более высоких порядков редки, трудны в реализации и всегда могут быть разбиты на совокупность бинарных связей.

**Четыре основных типа бинарных связей (стр. 18–19):**

1. **«Один к одному» (1:1)** (стр. 18)  
   Одному экземпляру объекта А соответствует один и только один экземпляр объекта В.  
   *Пример на стр. 18*: связь между персональной и служебной характеристикой работника (хранятся раздельно для защиты данных).  
   *Примечание*: на практике такой тип связи встречается редко, так как два объекта обычно можно объединить в один.

2. **«Один ко многим» (1:M)** (стр. 18–19)  
   Одному экземпляру объекта А соответствует несколько (много) экземпляров объекта В.  
   *Пример*: один клиент может сделать много покупок; один товар может быть завезён на склад многократно.

3. **«Многие к одному» (M:1)** (стр. 19)  
   Одному или нескольким экземплярам объекта А соответствует один экземпляр объекта В.  
   *Примечание*: связи «один ко многим» и «многие к одному» являются обратимыми, поэтому на практике обычно говорят о связи «один ко многим».

4. **«Многие ко многим» (M:N)** (стр. 19)  
   Одному экземпляру А соответствует несколько экземпляров В, и наоборот.  
   *Пример*: авторы и книги — книга может быть написана несколькими авторами, и один автор может написать много книг.  
   *Пример*: клиенты и товары — каждый товар может быть куплен несколькими клиентами, и один клиент может купить несколько товаров.

**Важные практические рекомендации (стр. 19):**
- При определении типа связи следует **перестраховаться** и выбрать более универсальный тип связи (1:1 → 1:M, 1:M → M:N), потому что в противном случае реализовать «недосмотренную» связь впоследствии будет невозможно.
- Но также не стоит всегда выбирать M:N, так как её реализация связана с рядом трудностей (введение дополнительных связующих объектов).

### ✅ Итог простыми словами

> **Связи между объектами** показывают, как объекты связаны друг с другом в БД.
> - **1:1** (один к одному): одному объекту соответствует ровно один другой. Встречается редко.
> - **1:M** (один ко многим): один объект связан со многими. Например, одна группа → много студентов.
> - **M:1** (многие к одному): то же самое, что и 1:M, но с другой стороны.
> - **M:N** (многие ко многим): многие объекты связаны со многими. Например, студенты и учебные предметы.

## Вопрос 12. Модель «сущность–связь» (ER-модель)

### 📚 Подробное раскрытие по материалу пособия

На стр. 22–26 (раздел 2.3) подробно описана ER-модель.

**Происхождение и назначение (стр. 22):**  
Модель «сущность–связь» (Entity-Relationship model, ER model) — это **высокоуровневая концептуальная модель данных**, созданная Питером Ченом в 1976 году в целях упрощения задачи проектирования БД.

**Основные понятия (стр. 22):**

1. **Сущность (entity)** — основной объект ER-модели. Может быть реально существующим (физическим) или абстрактным (концептуальным) объектом.  
   - Каждая сущность имеет **набор атрибутов**, представляющих её свойства.  
   - **Тип сущности** (имя) — совокупность сущностей с одинаковым набором атрибутов.  
   - **Экземпляр сущности** — конкретный объект с конкретными значениями атрибутов.

2. **Сильные и слабые типы сущностей (стр. 22)**  
   - *Сильный тип*: существование не зависит от других типов сущностей (родительский, владелец, доминантный).  
   - *Слабый тип*: существование зависит от другого типа сущности (дочерний, подчинённый, зависимый).

3. **Связи (relationships)** (стр. 22)  
   Между типами сущностей могут существовать связи. Как и для сущностей, различают **связи** и **типы связей**. Связи могут иметь собственные атрибуты.

4. **Ограничения на связи (стр. 22, 24–25)**  
   На связи накладываются ограничения по:
   - **показателю кардинальности** — тип связи (1:1, 1:M, M:1, M:N), определяемый бизнес-правилами предприятия;
   - **степени участия** — полное или частичное.

   *Степень участия (стр. 24–25)*:  
   - **Полное участие**: для существования объекта в одной сущности требуется существование связанного объекта в другой сущности. Отображается двойной линией.  
   - **Частичное участие**: объект может существовать без связи.

5. **Альтернативное обозначение (стр. 25)**  
   Допустимо использование пары чисел (min, max), где min — минимальное количество экземпляров (0 — частичное участие, >0 — полное), max — максимальное (N — отсутствие ограничений).

**Графическая нотация ER-модели (стр. 25–26, Рис. 5):**

| Элемент | Обозначение |
|---------|-------------|
| Тип сущности (сильный) | Прямоугольник |
| Тип сущности (слабый) | Двойной прямоугольник |
| Атрибут | Эллипс, соединённый линией |
| Производный атрибут | Эллипс с штриховой линией |
| Многозначный атрибут | Эллипс с двойной линией |
| Ключевой атрибут | Подчёркнутый текст в эллипсе |
| Составной атрибут | Эллипс, соединённый с другими эллипсами |

**Достоинства и недостатки ER-модели (стр. 26):**

*Достоинства:*
- Наличие удобной и выразительной диаграммной техники.
- Простота и лёгкость восприятия конечными пользователями.

*Недостатки:*
- Возможность появления **«ловушек»** при недостаточном определении связей.

**Ловушки (стр. 26–28, Рис. 6 и 7):**

1. **Ловушка разветвления** — возникает, когда две и более связи «один ко многим» разветвляются из одной сущности, и путь между сущностями указан неоднозначно.  
   *Пример (Рис. 6)*: схема не позволяет однозначно ответить на вопрос, к каким группам относятся студенты. Устраняется реорганизацией связей.

2. **Ловушка разрыва** — возникает между тремя и более сущностями, связанными отношениями «один ко многим» при наличии связи с частичным участием, когда предполагается связь, но пути между отдельными сущностями не существует.  
   *Пример (Рис. 7)*: схема не позволяет ответить, какое имущество числится за отделом. Устраняется введением дополнительной связи.

### ✅ Итог простыми словами

> **ER-модель** — это способ рисовать схему базы данных в понятных картинках.  
> - **Сущность** — это «объект» (например, Студент, Группа, Преподаватель).  
> - **Связь** — как объекты связаны (например, Студент учится в Группе).  
> - **Атрибуты** — свойства сущности (имя студента, номер группы).  
> Сильные сущности существуют сами по себе, слабые — только в составе других.  
> Главное достоинство — наглядность. Недостаток — можно ошибиться и получить «ловушки», когда схема не даёт однозначного ответа на простой вопрос.

## Вопрос 13. Типы сущностей (сильные/слабые) и типы связей (полное/частичное участие)

### 📚 Подробное раскрытие по материалу пособия

**Типы сущностей (стр. 22):**

- **Сильный тип сущности** — тип, существование которого **не зависит** от всех других типов сущностей. Сильные сущности называются родительскими, владельцами или доминантными.

- **Слабый тип сущности** — тип, существование которого **зависит** от какого-то другого типа сущности. Слабые сущности называются дочерними, подчинёнными или зависимыми.

**Степень участия в связи (стр. 24–25):**

- **Полное участие** — для существования объекта, участвующего в связи, требуется существование другого объекта.  
  Отображается **двойной линией** на диаграмме.  
  *Пример (стр. 24–25)*: если бизнес-правила требуют, чтобы любой служащий работал в некотором отделе, то экземпляр сущности «Служащий» будет существовать, только если он участвует в связи «Работает в» с сущностью «Отдел». Здесь участие сущности «Служащий» — полное.

- **Частичное участие** — объект может существовать и без участия в данной связи.  
  *Пример (стр. 25)*: только некоторые служащие могут руководить отделами. Значит, участие сущности «Служащий» в связи «Руководство» будет только частичным.

**Альтернативная форма обозначения (стр. 25, Рис. 5):**  
Пара чисел (min, max):
- min = 0 → частичное участие;
- min > 0 → полное участие;
- max = N → отсутствие ограничения на максимум;
- max = 1 → максимум один экземпляр.

*Пример из пособия (стр. 25)*: надпись (0,1) над связью «Руководство» означает, что любой служащий не обязан руководить отделом (min=0), и руководить двумя отделами одновременно он не может (max=1).

### ✅ Итог простыми словами

> - **Сильные сущности** существуют сами по себе (например, «Отдел»).  
> - **Слабые сущности** зависят от других (например, «Сотрудник» может существовать только в рамках какого-то отдела).  
> - **Полное участие** — объект не может существовать без связи (как сотрудник без отдела).  
> - **Частичное участие** — объект может существовать и без этой связи (не все сотрудники — руководители).  
> На схемах полное участие рисуют двойной линией, а частичное — обычной.

## Вопрос 14. Классические (инфологические) модели данных: иерархическая и сетевая

### 📚 Подробное раскрытие по материалу пособия

На стр. 27–30 (разделы 3.1 и 3.2) описаны классические модели данных. В пособии указано (стр. 27), что к числу классических (инфологических) моделей относятся **иерархическая, сетевая и реляционная**. Реляционная модель будет рассмотрена отдельно в вопросе 15.

**1. Иерархическая модель (стр. 27–28)**

*Историческая справка*: исторически первой была разработана и воплощена иерархическая модель данных. В 1960 г. была разработана БД для большой ЭВМ IBM System 360. Первой коммерческой СУБД стала IMS фирмы IBM.

*Структура* (стр. 27, Рис. 8): иерархическая модель данных представляется связным графом типа **дерева**, вершины (типы) которого расположены на разных иерархических уровнях. Одна вершина на самом верху называется **корнем** и не подчиняется ни одной вершине. Все остальные вершины связаны с одной и только одной вершиной, расположенной на более высоком уровне. Элементы в конце ветви, не имеющие порождённых, называются **листьями**. Потомков одного и того же типа называют **близнецами**.

*Связи*: иерархической модели присущи связи «один к одному» и «один ко многим». Связь «многие ко многим» не может быть реализована, так как вершина не может иметь более одного родителя.

*Пример* (стр. 28, Рис. 9): расписание занятий на факультете. Пособие отмечает, что иерархической модели данных присуще дублирование информации. В примере для каждой группы будут дублироваться записи расписания. При изменении расписания общих курсов придётся менять записи для всех групп. К тому же отсутствует связь расписания с преподавателями и преподавателей со студентами.

*Достоинства иерархической модели (стр. 28):*
- эффективность её реализации;
- хорошее использование памяти ЭВМ;
- неплохие показатели времени выполнения основных операций с данными.

*Недостатки иерархической модели (стр. 28):*
- невозможность установления связи M:N;
- сильная логическая зависимость данных (добавление или удаление вершины невозможно без разрушения всей структуры данных);
- избыточность информации;
- затруднённый поиск «снизу вверх» (приводит к простому перебору всех значений в БД).

**2. Сетевая модель (стр. 28–30)**

*Историческая справка*: впервые сетевые СУБД появились в 1970-е годы с появлением миникомпьютеров DEC. Примеры: IDMS, VAX_DBMS, db_VistaIII, CETb.

*Структура* (стр. 28–29, Рис. 10): если порождённый элемент имеет более одного исходного элемента, то такое отношение описывается в виде **сетевой структуры**. Сетевая модель данных позволяет отображать взаимосвязи элементов данных в виде **произвольного графа**, обобщая тем самым иерархическую модель. Данные представлены в виде записей и связей.

*Достоинства сетевой модели (стр. 29):*
- дублирование информации сильно уменьшается;
- большая гибкость (можно использовать не только предопределённые связи, но и добавлять новые в процессе работы);
- разрешены симметричные запросы (сверху вниз и снизу вверх), выполняющиеся по схожим алгоритмам.

*Пример (стр. 29, Рис. 11)*: сетевая модель расписания занятий. Здесь связь M:M между группами и преподавателями реализована в виде записей пересечения «расписание», указывающих день, время и номер аудитории. Между студентами и преподавателями также можно установить связь M:M с помощью записей пересечения «задание».

*Стандарт*: результатом развития сетевых моделей явился стандарт **CODASYL**. Основные языки программирования для создания сетевых и иерархических БД — PL/1 и COBOL.

*Недостатки сетевой модели (стр. 30):*
- недостаточно высокая скорость выполнения поиска;
- некоторая избыточность информации (для каждой записи дублируются ссылки на подчинённые и родительские элементы);
- жёсткость схемы БД (всё ещё слабая логическая независимость от данных);
- ослабление контроля целостности вследствие допустимости установления произвольных связей между записями.

### ✅ Итог простыми словами

> **Иерархическая модель** — данные организованы как дерево (как папки и подпапки на компьютере). Плюсы: быстро и эффективно. Минусы: нельзя сделать связь «многие ко многим», трудно искать снизу вверх, много повторов данных.
>
> **Сетевая модель** — данные организованы как произвольный граф (у записи может быть много «родителей»). Плюсы: меньше повторов, гибкость, можно искать в обе стороны. Минусы: сложность, скорость поиска ниже, слабый контроль целостности.

## Вопрос 15. Реляционная модель данных. Понятие отношения

### 📚 Подробное раскрытие по материалу пособия

**Основное определение (стр. 30):**  
В реляционной модели данных (РМД) информация хранится в одной или нескольких связанных таблицах. Реляционной называется база данных, в которой все данные, доступные пользователю, организованы в виде таблиц, а все операции над данными сводятся к операциям над этими таблицами. Отдельная таблица обычно представляет совокупность объектов, абстрактных концепций или событий одного типа. Каждая строка в таблице идентифицирует один объект группы.

**Свойства таблиц в реляционной модели (стр. 30):**

1. Каждая ячейка таблицы представляет собой **один атомарный элемент данных** (совокупность значений в одном столбце одной строки недопустима).
2. Все столбцы в таблице **однородные** — элементы столбца имеют одинаковую природу. Столбцам присвоены имена.
3. В таблице **нет двух одинаковых строк**.
4. Порядок размещения строк и столбцов в таблице может быть **произвольным**. При операциях строки и столбцы могут просматриваться в любом порядке безотносительно к их информационному содержанию.

Таблица, обладающая вышеперечисленными свойствами, является математическим двумерным множеством — **отношением** (стр. 30).

**Историческая справка (стр. 30–31):**  
В 1970 году инженер IBM Эдгар Ф. Кодд предложил модель данных, основанную на математических операциях исчисления отношений и реляционной алгебре. Основной структурной единицей этой модели было двумерное математическое множество «отношение» (relation), отсюда название.

**Терминология реляционной модели (стр. 31):**

| Математический термин | Аналог в таблице |
|----------------------|------------------|
| Отношение (relation) | Таблица |
| Атрибут (attribute) | Столбец (поле) |
| Кортеж (tuple) | Строка (запись) |
| Домен (domain) | Множество допустимых значений атрибута |
| Степень отношения | Количество атрибутов |
| Кардинальное число | Количество кортежей |

**Домен (стр. 31):**  
Домен — это множество всех возможных значений определённого атрибута отношения. Домен нельзя путать с атрибутом: атрибуты представляют различные интерпретации домена. Например, атрибуты «Имя клиента» и «Имя поставщика» берут значения из одного домена «Имена».

**Математическое определение отношения (стр. 32):**  
Пусть даны n множеств D₁, D₂, …, Dₙ (домены). Тогда **отношение R** есть множество упорядоченных кортежей ⟨d₁, d₂, …, dₙ⟩, где dₖ ∈ Dₖ. Иначе: n-арным отношением R (степени n) называют подмножество декартова произведения множеств D₁ × D₂ × … × Dₙ.

Отношение R состоит из:
- **заголовка (схемы)** — множество атрибутов (имён доменов);
- **тела** — множество кортежей, соответствующих заголовку.

**Различие отношения и таблицы (стр. 32–33):**  
Отношение — это абстрактный математический объект, а таблица — конкретное изображение этого объекта. В отношении строки и столбцы не могут быть упорядочены (это множество), а в таблице строки упорядочены сверху вниз, столбцы — слева направо. В таблице могут повторяться строки, а в отношении — нет.

### ✅ Итог простыми словами

> **Реляционная модель** — это способ организации данных в виде таблиц. Каждая таблица — это **отношение**: набор строк (кортежей) и столбцов (атрибутов) с жёсткими правилами: в каждой ячейке — одно значение, все данные в столбце однотипные, нет одинаковых строк, порядок строк и столбцов не важен. Автор модели — Эдгар Кодд (1970 г.).

## Вопрос 16. Назначение первичных и внешних ключей

### 📚 Подробное раскрытие по материалу пособия

**Первичный ключ (primary key) (стр. 31):**

> *«Первичным ключом называется атрибут отношения, однозначно идентифицирующий каждый из его кортежей»*.

Ключ может быть **составным (сложным)**, т. е. состоять из нескольких атрибутов.

**Потенциальный ключ (стр. 31):**  
Потенциальный ключ — это подмножество атрибутов отношения, обладающее двумя свойствами:
1. **Свойством уникальности** — нет одинаковых кортежей с теми же значениями потенциальных ключей.
2. **Свойством неизбыточности** — никакое из подмножеств потенциального ключа не обладает свойством уникальности.

Один потенциальный ключ выбирается в качестве **первичного**, остальные остаются **потенциальными (альтернативными)** ключами.

**Важность ключей в реляционной теории (стр. 32):**  
Первичные (в общем случае потенциальные) ключи являются **детерминантами отношений** и служат для **адресации кортежей**. Указав значение потенциального ключа, мы гарантированно получим не более одного кортежа.

**Внешний ключ (foreign key) (стр. 31):**

> *«Внешний ключ — это атрибут подчинённого отношения, который используется для установления связи с базовым отношением. Он содержит значения, всегда совпадающие со значениями первичного ключа базового отношения»*.

**Правила внешних ключей для обеспечения ссылочной целостности (стр. 42):**

1. **Ограничение обновления и удаления**:
   - При работе с базовой таблицей запрещает удаление записей, содержащих связанные записи в подчинённых таблицах, и изменение значений их первичных ключей. При нарушении — ошибка «Master has detail records».
   - При работе с подчинённой таблицей запрещает установку внешнего ключа в значение, не существующее в базовой таблице. При нарушении — ошибка «Key violation».

2. **Каскадное обновление и каскадное удаление**:
   - При изменении значения базового ключа значения внешних ключей подчинённых таблиц изменяются соответственно (каскадное обновление).
   - При удалении записи из базовой таблицы все связанные записи из подчинённых таблиц также удаляются (каскадное удаление).

**Как тип связи определяется через ключи (стр. 40–41):**

| Тип связи | Участвующие атрибуты |
|-----------|----------------------|
| 1:1 | С обеих сторон участвуют ключевые атрибуты (дублирование исключено) |
| 1:M | С одной стороны — ключевой атрибут, с другой — неключевой |
| M:N | Реализуется через дополнительную таблицу с внешними ключами |

**Цели использования ключей в реляционных БД (стр. 53):**

1. Исключение дублирования кортежей и, как следствие, адресация кортежей.
2. Организация связывания отношений.

### ✅ Итог простыми словами

> **Первичный ключ** — это уникальный номер записи в таблице (как паспорт у человека). По нему можно найти ровно одну запись.
>
> **Внешний ключ** — это поле в одной таблице, которое ссылается на первичный ключ другой таблицы. Как «ссылка» в документе на другой документ. Благодаря этому таблицы связываются друг с другом.
>
> Ключи нужны, чтобы не было повторов и чтобы связывать таблицы.

## Вопрос 17. Обеспечение целостности в реляционной модели

### 📚 Подробное раскрытие по материалу пособия

На стр. 41–43 (раздел 3.3.3) подробно описаны **четыре типа ограничителей целостности** для реляционной БД:

**1. Ограничители домена (domain constraints) (стр. 42)**  
Запрещают ввод значений атрибутов, не принадлежащих домену. К ним относятся:
- задание типа данных;
- маска ввода;
- задание списка значений;
- задание диапазона значений.

**2. Ограничители ключей (key constraints) (стр. 42)**  
Обеспечивают уникальность значений первичных (в общем случае — потенциальных) ключей. Обеспечивается применением уникальных индексов.

**3. Ограничители записи (entity constraints) (стр. 42–43)**  
Запрещают NULL значения для первичных ключей и важных в семантическом смысле полей. Иначе будет нарушено требование идентифицируемости записи — можно получить запись, имеющую первичный ключ, но целиком состоящую из пустых значений.

**4. Ссылочная целостность (Declarative Referential Integrity, DRI) (стр. 41–42)**  

> *«Реляционная БД не должна содержать несогласованных значений внешних ключей, т. е. таких значений, для которых не существует отвечающих им значений соответствующих потенциальных ключей базовых таблиц»*.

*Пример (стр. 41)*: если в БД уже есть клиент с номером 10 и товар с номером 25, только тогда можно внести запись о покупке клиентом 10 товара 25 в таблицу «Заказы». Нельзя удалить клиента с номером 25, если в БД есть хотя бы одна запись о сделанном им заказе — иначе согласованность внешних ключей нарушится.

**Два правила внешних ключей (стр. 42):**

| Правило | Действие | Ошибка |
|---------|----------|--------|
| Ограничение обновления/удаления | Нельзя удалить/изменить запись в базовой таблице, если есть связанные записи в подчинённых | «Master has detail records» |
| Ограничение вставки | Нельзя вставить внешний ключ со значением, отсутствующим в базовой таблице | «Key violation» |

| Альтернативное правило | Действие |
|------------------------|----------|
| Каскадное обновление | При изменении базового ключа автоматически меняются внешние ключи в подчинённых таблицах |
| Каскадное удаление | При удалении записи из базовой таблицы удаляются все связанные записи из подчинённых |

### ✅ Итог простыми словами

> В реляционной БД целостность обеспечивается четырьмя способами:
> 1. **Доменные ограничения** — проверяют, что данные правильного типа и в допустимых пределах.
> 2. **Ключевые ограничения** — не дают создать две записи с одинаковым первичным ключом.
> 3. **Ограничения записи** — требуют заполнить важные поля (первичный ключ не может быть пустым).
> 4. **Ссылочная целостность** — не позволяют создать «битую ссылку» (например, заказ без клиента). Есть два режима: запрет на удаление/изменение родителя («строгий») или автоматическое удаление/обновление детей («каскадный»).

## Вопрос 18. Операции реляционной алгебры

### 📚 Подробное раскрытие по материалу пособия

На стр. 33–36 (раздел 3.3.1) описана **реляционная алгебра**.

**Определение (стр. 33):**  
Реляционная алгебра — это **замкнутая система операций** над отношениями в реляционной модели данных. Результатом каждой реляционной операции является **отношение** (свойство **замкнутости**).

Выделяют **восемь основных реляционных операторов**, объединённых в две группы (стр. 33):

**Группа 1. Теоретико-множественные операторы:**

1. **Объединение (стр. 33–34)**  
   Объединением двух совместимых по типу отношений A и B называется отношение с тем же заголовком, что у A и B, и телом, состоящим из кортежей, принадлежащих A, или B, или обоим отношениям. Повторяющиеся кортежи исключаются.

2. **Пересечение (стр. 34)**  
   Пересечением двух совместимых по типу отношений A и B называется отношение с тем же заголовком, что у A и B, и телом, состоящим из кортежей, принадлежащих одновременно обоим отношениям A и B.

3. **Вычитание (стр. 34)**  
   Вычитанием двух совместимых по типу отношений A и B называется отношение с тем же заголовком, что у A и B, и телом, состоящим из кортежей, принадлежащих A и не принадлежащих B.

4. **Декартово произведение (стр. 34)**  
   Декартовым произведением двух отношений A(A₁,…,Aₙ) и B(B₁,…,Bₘ) называется отношение, заголовок которого — сцепление заголовков A и B: (A₁,…,Aₙ,B₁,…,Bₘ), а тело состоит из кортежей, являющихся сцеплением кортежей A и B: (a₁,…,aₙ,b₁,…,bₘ), таких что (a₁,…,aₙ) ∈ A, (b₁,…,bₘ) ∈ B. Получаются все возможные сочетания кортежей из A и B.

**Группа 2. Специальные реляционные операторы:**

5. **Проекция (вертикальная селекция) (стр. 34–35)**  
   Операция, при которой из отношения выделяются атрибуты только из указанных доменов — из таблицы выбираются только нужные столбцы. Если при этом получается несколько одинаковых кортежей, в результирующем отношении остаётся только по одному экземпляру.

6. **Выборка (горизонтальная селекция) (стр. 35)**  
   Операция, которая выделяет множество строк в таблице, удовлетворяющих заданному условию. Условием может быть любое логическое выражение.

7. **Соединение (стр. 35–36)**  
   Операция, обратная проекции. Создаёт новое отношение из двух существующих путём конкатенации кортежей, в которых совпадают значения заданных атрибутов.  
   *Синтаксис*: соединение можно представить как декартово произведение + выборка с условием равенства.

   *Три разновидности соединения (стр. 35–36):*
   - **Θ-соединение** (общее соединение) — соединение по условию c (где c — логическое выражение).
   - **Эквисоединение** — частный случай Θ-соединения, когда Θ — знак равенства: A [X = Y] B.
   - **Естественное соединение (NATURAL JOIN)** — соединение по всем атрибутам с одинаковыми именами, с удалением дублирующего столбца.

8. **Деление (стр. 34)**  
   В пособии указано, что операция деления не нашла широкого применения и подробно не рассматривается.

**Выразимость операций друг через друга (стр. 36):**

| Операция | Выражается через |
|----------|------------------|
| Соединение | Декартово произведение + выборка |
| Естественное соединение | + проекция |
| Пересечение | A ∩ B = A \ (A \ B) |
| Деление | A ÷ B = A[X] \ ((A[X] × B) \ A)[X] |

**Примитивные операции (не выражаются друг через друга):** объединение, вычитание, декартово произведение, выборка, проекция.

**Ограничения реляционной алгебры (стр. 36):**  
Некоторые типы запросов нельзя выразить средствами реляционной алгебры:
- запросы, требующие дать в ответе список атрибутов, удовлетворяющих определённым условиям;
- построение транзитивного замыкания отношений;
- построение кросс-таблиц.

### ✅ Итог простыми словами

> **Реляционная алгебра** — это 8 операций для работы с таблицами:
> - **Объединение** — берём все строки из двух таблиц (как в математике).
> - **Пересечение** — берём строки, которые есть в обеих таблицах.
> - **Вычитание** — из одной таблицы убираем строки, которые есть в другой.
> - **Декартово произведение** — «скрещиваем» каждую строку из А с каждой строкой из Б (все возможные пары).
> - **Проекция** — оставляем только нужные столбцы.
> - **Выборка** — оставляем только нужные строки по условию.
> - **Соединение** — соединяем две таблицы по совпадающим значениям (как VLOOKUP в Excel).
> - **Деление** — редко используется.

## Вопрос 19. Нормализация и денормализация данных. Понятие нормальных форм

### 📚 Подробное раскрытие по материалу пособия

**Нормализация данных (стр. 43):**

> *«Процесс нормализации данных позволяет решить вопрос о наиболее эффективной их структуре, обладающей минимальной избыточностью»*.

**Задачи нормализации (стр. 43):**
a) исключить ненужное повторение данных;
b) обеспечить быстрый доступ к данным;
c) обеспечить целостность данных.

**Основные принципы (стр. 43):**
- Теория нормализации оперирует с **пятью нормальными формами** и одной промежуточной.
- На практике обычно руководствуются только первыми тремя нормальными формами.
- Процесс проектирования — итерационный, последовательное применение правил от низшей к высшей.
- Каждая следующая нормальная форма уменьшает избыточность и сохраняет свойства всех предыдущих.
- Перевод в следующую форму осуществляется методом **«декомпозиции без потерь»** — разбиением исходной таблицы на несколько связанных.
- Декомпозиция должна обеспечить равенство результатов выборок из исходного отношения и выборок, основанных на совокупности полученных отношений.

**Аномалии обновления (стр. 44):**

1. **Аномалии добавления** — информацию в таблицу нельзя поместить, пока она не полная. Пример: нельзя добавить новый товар без покупателя.
2. **Аномалии удаления** — при удалении информации может пропасть и другая, не связанная напрямую. Пример: при удалении последней продажи товара теряются сведения о нём.
3. **Аномалии модификации** — изменение значения поля в одной записи может потребовать просмотра всей таблицы.

**Правила нормальных форм (стр. 45–51):**

**Первая нормальная форма (1НФ) (стр. 45)** — таблица находится в 1НФ, если:
1. Таблица не имеет повторяющихся записей.
2. Все атрибуты должны быть **простыми (скалярными)** — в каждой ячейке одно значение, не массив и не перечисление.

*Как привести к 1НФ*:
- Ввести ключевое поле (табельный номер), однозначно определяющее остальные поля.
- Составные поля разложить на простые (например, «Адрес» → «Страна», «Город», «Улица»).
- Многозначные поля вынести в отдельные таблицы.

**Вторая нормальная форма (2НФ) (стр. 46)** — таблица находится в 2НФ, если:
1. Она удовлетворяет условиям 1НФ.
2. Любое неключевое поле должно однозначно и **полно** идентифицироваться значением первичного ключа (не должно зависеть от части составного ключа — **полная функциональная зависимость**).

*Пример нарушения*: в таблице с ключом (Код_клиента, Код_товара, Дата_заказа) поля «Фамилия», «Имя» зависят только от «Код_клиента» — нарушение 2НФ. Их нужно вынести в отдельную таблицу «Клиенты».

**Третья нормальная форма (3НФ) (стр. 47)** — таблица находится в 3НФ, если:
1. Она удовлетворяет условиям 2НФ.
2. Ни одно неключевое поле не должно идентифицироваться с помощью другого неключевого поля (отсутствие **транзитивных зависимостей**).

*Пример нарушения*: в таблице «Товары» поле «Размещение» зависит от поля «Категория» (не от ключа «Код_товара»). Нужно создать отдельную таблицу «Категории».

**Нормальная форма Бойса – Кодда (BCNF) (стр. 49)** — усиленная 3НФ. Таблица находится в BCNF, если:
1. Удовлетворяет требованиям 3НФ.
2. Отсутствуют зависимости атрибутов составного ключа от неключевых атрибутов (каждый детерминант должен быть потенциальным ключом). Если отношение имеет единственный потенциальный ключ, то 3НФ и BCNF эквивалентны.

**Четвёртая нормальная форма (4НФ) (стр. 49–50)** — отношение находится в 4НФ, если:
1. Оно находится в BCNF.
2. В нём отсутствуют **нетривиальные многозначные зависимости**.

*Тривиальная многозначная зависимость*: Y ⊂ X или Y ∪ X = R (не нарушает 4НФ).  
*Нетривиальная многозначная зависимость*: требует дальнейшей декомпозиции, чаще всего сразу на три отношения.

**Пятая нормальная форма (5НФ) (стр. 50–51)** — основывается на устранении **зависимостей объединения**, возникающих после декомпозиции при устранении многозначных зависимостей. Может возникнуть ситуация, когда таблицу нельзя разбить на две так, чтобы после объединения получить исходный результат, но можно разбить на три и более.

**Резюме по нормальным формам (стр. 52):**

| Нормальная форма | Что устраняет |
|------------------|----------------|
| 5НФ | зависимости объединения |
| 4НФ | многозначные зависимости |
| 3НФ | транзитивные зависимости |
| 2НФ | частичные функциональные зависимости |
| 1НФ | составные и многозначные атрибуты (требует соответствия требованиям к отношениям) |

**Денормализация (стр. 51):**

> *«Денормализация — это намеренное приведение структуры базы данных в состояние, не соответствующее критериям нормализации, обычно проводимое с целью ускорения операций чтения из базы данных за счёт добавления избыточных данных»*.

*Когда применяется*: если в запросах выборки, основанных на большом числе отношений, операция соединения (JOIN) выполняется неприемлемо долго (обычно если требуется объединять более 5–6 отношений).

*Предупреждение*: денормализация всегда выполняется за счёт повышения риска нарушения целостности данных при операциях модификации. Денормализацию следует проводить в крайнем случае.

*Виды денормализации (стр. 51):*
- **Восходящая** — перенос информации из подчинённого отношения в родительское.
- **Нисходящая** — перенос информации из родительского отношения в подчинённое.
- **Разбиение одного отношения на два** — обычно когда запись имеет большую длину за счёт атрибутов большого объёма (BLOB, графика), которые редко используются.

**Правила нормализации (стр. 52):**
1. Разрабатывайте схему данных так, чтобы можно было легко её объяснить (не комбинируйте атрибуты независимых объектов, не создавайте сложные связи).
2. Разрабатывайте схему данных так, чтобы исключить возможность появления аномалий обновления.
3. Разрабатывайте схему данных так, чтобы в связях участвовали только первичные (потенциальные) и внешние ключи (избежать подложных записей).

**Важное замечание (стр. 52):** требование, чтобы все таблицы были нормализованы, **не является обязательным**. Вполне можно обойтись без некоторых таблиц.

**Итоговое определение реляционной БД (стр. 52):**  
Реляционная база данных — это совокупность **нормализованных отношений**.

### ✅ Итог простыми словами

> **Нормализация** — это процесс «причёсывания» таблиц, чтобы убрать лишние повторы и аномалии. Проходит по шагам:
> - **1НФ**: в каждой ячейке — одно значение, нет повторяющихся строк.
> - **2НФ**: нет зависимости полей от части составного ключа.
> - **3НФ**: нет зависимости неключевых полей от других неключевых полей.
> - **BCNF, 4НФ, 5НФ** — более строгие формы для сложных случаев.
>
> **Денормализация** — наоборот, специальное добавление повторов, чтобы ускорить чтение (но тогда сложнее обновлять). Делают только когда скорость чтения критически важна.

## Вопрос 20. Нереляционные (NoSQL) базы данных

### 📚 Подробное раскрытие по материалу пособия

На стр. 57–61 (раздел 3.1) описаны NoSQL базы данных.

**Определение и происхождение термина (стр. 57):**  
NoSQL (англ. not only SQL, «не только SQL») — термин, обозначающий ряд подходов, направленных на реализацию хранилищ баз данных, имеющих существенные отличия от реляционной модели с доступом к данным средствами языка SQL. В семейство NoSQL **не включаются** иерархические, сетевые, многомерные и объектно-ориентированные модели данных.

**Причины появления NoSQL (стр. 57–58):**

1. **Масштабируемость (scalability) и доступность (availability)** — реляционные БД хорошо масштабируются только на единственном сервере. Когда ресурсы сервера заканчиваются, требуются другие серверы и распределение нагрузки. При увеличении серверов до сотни или тысячи сложность распределённой реляционной системы возрастает на порядок. Попытка держать данные согласованными на всех серверах приводит к резкому падению производительности.

2. **Требования облачных сервисов** — популярные облачные сервисы требуют масштабируемого хранилища данных.

3. **Снижение производительности больших реляционных БД из-за JOIN** — в реляционной модели согласно правилам нормализации приходится выполнять декомпозицию логической сущности со сложной структурой. В результате постоянно требуется операция соединения (JOIN) данных из нескольких таблиц. NoSQL БД оперируют с сущностями как с целостными объектами. Главное правило NoSQL БД — **минимум объединений между таблицами**.

**Теорема CAP (стр. 60):**  
В распределённых вычислениях можно обеспечить только два из трёх свойств: **согласованность данных (Consistency)**, **доступность (Availability)** или **устойчивость к разделению (Partition tolerance)**.

**BASE вместо ACID (стр. 59):**  
В NoSQL вместо ACID главенствующим является набор свойств **BASE**:

| Свойство | Расшифровка |
|----------|-------------|
| Basic availability | Каждый запрос гарантированно завершается (успешно или безуспешно) |
| Soft state (гибкое состояние) | Состояние системы может меняться со временем, даже без ввода новых данных, для достижения согласования |
| Eventual consistency (согласованность в конечном счёте) | Данные могут быть некоторое время рассогласованы, но в итоге становятся согласованными |

**Другие отличия NoSQL от реляционных БД (стр. 58–59):**

1. **Слабая типизация (schemaless)** — структура данных не регламентирована. В отдельной записи или документе можно добавить произвольное поле без предварительного изменения структуры всей таблицы. Поле можно добавить динамически, в процессе работы с БД. Это полезно для разреженных данных (каталоги товаров, где у разных типов товаров разные поля).

   *Проблемы*: добавленное поле отсутствует у других объектов. Решения: обойти все документы (большие накладные расходы) или проверять наличие поля в коде приложения (замедляет работу).

2. **Данные как агрегаты (aggregates)** — операция JOIN не требуется, так как все данные одной сущности хранятся вместе.

**Типы NoSQL хранилищ (стр. 60–61):**

| Тип | Описание | Примеры | Применение |
|-----|----------|---------|------------|
| **Ключ-значение** (key-value store) | Простейшее хранилище, ключ для доступа к значению. Вся информация записи хранится вместе с ней | Berkeley DB, MemcacheDB, Redis, Amazon DynamoDB | Хранение изображений, файловые системы, кэши |
| **Семейств колонок** (column database) | Данные в виде разреженной матрицы (строки и столбцы — ключи). Колонка: имя + значение + метка времени | Apache HBase, Apache Cassandra, SimpleDB | Веб-индексирование, большие наборы данных с низкими требованиями к согласованности |
| **Документо-ориентированные** (document store) | Хранение иерархических структур данных в документах (JSON-подобных объектах), организованных в коллекции | CouchDB, MongoDB | Системы управления содержимым, документный поиск, каталогизация товаров |
| **Графовые** (graph database) | Для данных с большим количеством связей | OrientDB | Социальные сети, выявление мошенничества |

**Основные признаки NoSQL баз данных (стр. 61):**
1. Не используется SQL.
2. Неструктурированное хранение данных (schemaless).
3. Представление данных в виде агрегатов (aggregates).
4. Слабые ACID свойства транзакций или требования BASE.
5. Возможность создания распределённых систем без совместно используемых ресурсов (share nothing).

**Достоинства и недостатки NoSQL (по тексту пособия):**

*Достоинства:*
- Высокая масштабируемость (до сотен и тысяч серверов).
- Высокая доступность.
- Быстрое время отклика (за счёт отказа от JOIN и строгой согласованности).
- Гибкость схемы (можно динамически добавлять поля).

*Недостатки:*
- Отсутствие строгой согласованности (согласованность в конечном счёте).
- Ослабленная целостность данных.
- Сложности с обеспечением безопасности.
- Отсутствие стандартов.

**Перспективы NoSQL и реляционных БД (стр. 61):**  
Несмотря на всплеск интереса к NoSQL, NoSQL БД не смогут полностью вытеснить реляционные БД. Последние, обладая гораздо более высокой безопасностью хранения данных и поддержкой их согласованности, имеют все шансы на долгое существование. Проблемы с масштабируемостью реляционных СУБД несколько преувеличены — они начинают возникать на десятках и сотнях терабайт данных, которые есть только у нескольких сотен компаний в мире.

### ✅ Итог простыми словами

> **NoSQL** — это «не только SQL», то есть базы данных, которые устроены иначе, чем обычные таблицы. Появились, потому что обычные базы данных плохо работают на тысячах серверов в интернете (соцсети, магазины).
>
> Главные отличия:
> - **BASE** вместо **ACID** — можно временно ошибаться, лишь бы быстро и доступно.
> - **Schemaless** — можно добавлять поля на лету.
> - **Минимум JOIN** — данные хранятся не по нормальным формам, а «как есть».
>
> Типы NoSQL:
> - **Ключ-значение** (как огромный словарь).
> - **Документо-ориентированные** (как папка с JSON-файлами, MongoDB).
> - **Колоночные** (для очень больших данных, как в Google).
> - **Графовые** (для соцсетей, где много связей).
>
> NoSQL не заменит полностью обычные БД — обычные БД надёжнее и безопаснее.

# Глава 4

## Вопрос 21. Этапы проектирования базы данных

### 📚 Подробное раскрытие по материалу пособия

На стр. 80–84 (раздел 4.1) и Рис. 16 (стр. 81) описаны **девять основных этапов проектирования БД**.

**Требования к «хорошей» БД (стр. 80):**

Прежде чем перейти к этапам, в пособии формулируются требования, которым должна удовлетворять хорошая БД:

1. Должна удовлетворять информационным потребностям пользователей (организаций) и по структуре и содержанию соответствовать решаемым задачам.
2. Должна обеспечивать получение требуемых данных за приемлемое время, т. е. отвечать требованиям производительности.
3. Должна легко расширяться при реорганизации предметной области.
4. Должна легко изменяться при изменении программной и аппаратной среды.
5. Корректные данные, загруженные в БД, должны оставаться корректными (данные должны проверяться на корректность при их вводе).

**Этапы проектирования (стр. 80–84):**

**1-й этап. Планирование разработки БД (стр. 80)**  
На этом этапе выделяется наиболее эффективный способ реализации этапов жизненного цикла системы.

**2-й этап. Определение требований к системе (стр. 80–81)**  
Производится определение диапазона действий и границ приложения БД, а также сбор и анализ требований пользователей. Составляется словесное описание предметной области.

**3-й этап. Инфологическое моделирование (концептуальное проектирование) (стр. 81–82)**  
Заключается в разработке **концептуальной модели** предметной области, представляющей объекты и их взаимосвязи без указания способов их физического хранения. Инфологическое проектирование прежде всего связано с попыткой представления семантики (смысла) предметной области в модели БД.

*Почему это важно* (стр. 81): конкретные логические модели данных (например, реляционная) не позволяют отобразить семантику во всех деталях, поэтому на этом этапе лучше пользоваться более общими моделями (например, моделью «сущность–связь»).

*Уточнение задач (стр. 81–82)*:

a) **Уточнение задачи** — перед началом работы разработчик должен явно представлять, что он будет разрабатывать. Для больших БД под заказ представлений может быть мало или они поверхностны. Не стоит сразу начинать с определения таблиц, полей и связей — это приведёт к переделке.

b) **Уточнение последовательности выполнения задач** — лучше объединить основные задачи в группы и упорядочить их по порядку выполнения.

c) **Анализ данных** — после определения списка задач для каждой задачи составить полный перечень требуемых данных. Затем приступать к выделению объектов, атрибутов и связей.

**4-й этап. Даталогическое проектирование (логическое моделирование) (стр. 82)**  
Заключается в разработке **логической модели** предметной области. Инфологическая модель преобразуется в даталогическую, соответствующую конкретной СУБД. Этап начинается с выбора модели данных. При выборе модели важную роль играют её простота, наглядность и сравнение естественной структуры данных с моделью. Но часто этот выбор определяется успехом (или наличием) той или иной СУБД — разработчик выбирает СУБД, а не модель данных.

*Важное замечание*: возможно, что отображённые в концептуальной модели взаимосвязи или атрибуты окажутся нереализуемыми средствами выбранной СУБД. Это потребует изменения концептуальной модели. Версия концептуальной модели, которая может быть обеспечена конкретной СУБД, называется **логической моделью**.

Иногда процесс определения концептуальной и логической моделей называют определением структуры данных. Для концептуального моделирования чаще всего используется модель «сущность–связь» и методология **IDEF1X**.

**5-й этап. Построение физической модели (стр. 82–83)**  
Физическая модель определяет **размещение данных, методы доступа и технику индексирования**. На этапе физического проектирования мы привязываемся к конкретной СУБД и расписываем схему данных более детально, с указанием типов, размеров полей и ограничений. Кроме разработки таблиц и индексов, производится также определение основных запросов.

*Две противоположные по сути задачи (стр. 83)*:

| Задача | Что требуется |
|--------|---------------|
| 1. Минимизация места хранения данных | Экономия дискового пространства |
| 2. Максимальная производительность, целостность и безопасность | Создание индексов, журналирование, резервное копирование, резервирование места под временные объекты (приводит к увеличению размера БД) |

**6-й этап. Оценка физической модели (стр. 83)**  
Проводится оценка эксплуатационных характеристик: эффективность выполнения запросов, скорость поиска, правильность и удобство выполнения операций, целостность данных и эффективность расхода ресурсов компьютера. При неудовлетворительных характеристиках возможен возврат к пересмотру физической и логической моделей, выбору СУБД и типа компьютера.

**7-й этап. Реализация БД (стр. 83)**  
При удовлетворительных эксплуатационных характеристиках можно перейти к созданию **макета приложения** — набору основных таблиц, запросов, форм и отчётов. Этот предварительный макет можно продемонстрировать заказчику и получить его одобрение перед детальной реализацией.

**8-й этап. Тестирование и оптимизация (стр. 83)**  
Обязательным этапом является тестирование и оптимизация разработанного приложения.

**9-й этап (заключительный). Сопровождение и эксплуатация (стр. 83)**  
Так как выявить и устранить все ошибки на этапе тестирования не получается, этап сопровождения является обычным для БД.

**Два основных подхода к проектированию схемы данных (стр. 83–84):**

| Подход | Суть | Применение | Пример |
|--------|------|------------|--------|
| **Восходящий** (bottom-up) | Начинается с нижнего уровня — определения атрибутов, которые группируются в объекты на основе анализа связей | Для относительно небольших БД | Нормализация таблиц |
| **Нисходящий** (top-down) | Начинается с определения нескольких высокоуровневых сущностей и связей, затем детализация | Для БД с数百 до数千 атрибутов | Модель «сущность–связь» |

На практике эти подходы обычно **комбинируются** (смешанный подход).

**Каноническая форма логической модели (стр. 84):**  
Логическая модель считается представленной в канонической форме, если она правильно структурирована:

1. Сложные связи преобразованы в совокупность простых.
2. Развязаны отношения M:N, участвующие в простых связях.
3. Логическая модель упорядочена по уровням иерархии (на верхнем уровне — сильные сущности, затем подчинённые и т.д.).  
*Примечание*: последнее требование обязательно при проектировании автоматизированных систем управления (АСУ), а не БД.

### ✅ Итог простыми словами

> Проектирование БД проходит 9 этапов:
> 1. **Планирование** — как будем делать.
> 2. **Сбор требований** — что нужно заказчику.
> 3. **Инфологическое моделирование** — рисуем схему «сущности и связи» (без привязки к компьютеру).
> 4. **Даталогическое проектирование** — переводим схему в модель конкретной СУБД (например, реляционную).
> 5. **Физическое проектирование** — решаем, где и как хранить данные, какие индексы сделать (тут два желания: экономить место vs. сделать быстро).
> 6. **Оценка** — проверяем, быстро ли работает.
> 7. **Реализация** — делаем рабочий макет.
> 8. **Тестирование** — ищем ошибки.
> 9. **Сопровождение** — поддерживаем и чиним после запуска.
>
> Есть два подхода: **снизу вверх** (от полей к таблицам, через нормализацию) — для маленьких БД, и **сверху вниз** (от главных сущностей к деталям) — для больших. Обычно их смешивают.

## Вопрос 22. Назначение и классификация средств автоматизированной разработки приложений (CASE-системы)

### 📚 Подробное раскрытие по материалу пособия

На стр. 89–91 (раздел 4.3) описаны CASE-системы.

**Определение CASE-систем (стр. 89):**

> *«CASE (Computer Aided Software Engineering) системы — это программные средства, поддерживающие процессы создания и сопровождения информационных систем, такие как анализ и формулировка требований, проектирование приложений, генерация кода, создание развёртываемых систем (инсталляторов), тестирование, управление конфигурацией и проектом»*.

**Основная цель CASE-систем (стр. 89):**  
Отделить процесс проектирования программного обеспечения от его кодирования и последующих этапов разработки (тестирование, документирование и т.д.), а также **автоматизировать весь процесс создания программных систем**.

**Возможности CASE-систем при проектировании БД (стр. 89–90):**

- На этапе концептуального проектирования обычно используется модель «сущность–связь» как способ концептуального проектирования данных.
- Результатом проектирования (до недавнего времени) почти всегда являлась реляционная модель данных. В последнее время с появлением объектно-ориентированной модели данных конечным результатом всё чаще становятся объектно-ориентированная и объектно-реляционная модели.

**Две ключевые функции CASE-систем (стр. 90):**

| Функция | Направление | Суть |
|---------|-------------|------|
| **Forward engineering** (прямое проектирование) | От логической модели к физической | Автоматическая генерация структуры физической БД на основе логической модели данных |
| **Reverse engineering** (обратное проектирование) | От физической БД к логической модели | Генерация из функционирующей физической БД соответствующей ей логической модели данных |

При этом поддерживается синхронизация обеих моделей данных.

**Язык UML (стр. 90–91):**  
Для автоматизации проектирования **объектно-ориентированных баз данных** CASE-системы предоставляют специальный язык **UML (Unified Modeling Language)** — промышленный объектно-ориентированный стандарт моделирования. Его составляющими являются языки OMT (Object Modeling Technique) и OOSE (Object-Oriented Software Engineering). Большую роль в создании UML сыграл консорциум **OMG (Object Management Group)**.

**Классификация CASE-систем (стр. 91):**

**1. По ориентации:**

- CASE-системы, предназначенные для **анализа предметной области**.
- CASE-системы, предназначенные для **проектирования БД**.
- CASE-системы, предназначенные для **полной разработки приложений** (к числу последних можно отнести MS Visual Studio).

**2. По функциональной полноте:**

- Системы, предназначенные для решения **отдельных задач** проектирования.
- **Интегрированные системы**, поддерживающие **весь цикл разработки**.

**3. По типу используемой модели:**

- **Структурные**.
- **Объектно-ориентированные**.
- **Комбинированные**.

### ✅ Итог простыми словами

> **CASE-системы** — это программы, которые помогают проектировать базы данных и приложения. Они умеют:
> - Рисовать схему БД (сущности и связи).
> - Автоматически создавать таблицы по схеме (forward engineering).
> - Наоборот, по готовой БД восстанавливать схему (reverse engineering).
>
> **Классификация**:
> - По **цели**: для анализа, для проектирования БД, для полной разработки.
> - По **объёму**: узкие (одна задача) или интегрированные (всё в одном).
> - По **модели**: структурные, объектно-ориентированные или смешанные.
>
> Для объектно-ориентированных БД используется специальный язык **UML**.

## Вопрос 23. Методология IDEF1X

### 📚 Подробное раскрытие по материалу пособия

На стр. 84–89 (раздел 4.2) подробно описана методология IDEF1X.

**Определение и происхождение (стр. 84):**

> *«Методология IDEF1X — один из подходов к семантическому моделированию данных, основанный на модели сущность–связь и позволяющий построить модель данных, эквивалентную реляционной модели в третьей нормальной форме»*.

IDEF1X является одним из стандартов моделирования сложных систем, предложенных в семействе методологий **IDEF (Integrated DEFinition)**. Всего было предложено 15 стандартов (IDEF0 – IDEF14).  
**IDEF0** — методология функционального моделирования (система как набор взаимосвязанных функциональных блоков).  
**IDEF1X** (IDEF1 eXtended) — расширение стандарта IDEF1, разработанное Т. Рэмей.

В настоящее время в рамках этой методологии предложено несколько вариантов графического отображения объектов **(нотаций)**, в большей степени отличающихся по виду представления связей между таблицами.

**Основные понятия IDEF1X (стр. 84–86):**

1. **Сущности (entity)** — представляют объекты реального мира (аналогично модели сущность–связь). Каждая сущность имеет имя и набор атрибутов.

2. **Типы сущностей (стр. 84–85):**
   - **Независимые сущности** — изображаются прямоугольником с прямыми углами.
   - **Зависимые сущности** — изображаются прямоугольником с закруглёнными углами.

3. **Атрибуты (стр. 84–85):**
   - **Ключевые атрибуты** — записываются вверху списка атрибутов и отделяются от неключевых атрибутов линией.
   - **Необязательные к заполнению атрибуты** — помечаются символом (O — optional).

**Типы связей в IDEF1X (стр. 85–87):**

1. **По идентификации (стр. 85):**
   - **Идентифицирующая связь** — между независимой и зависимой сущностями. Внешний ключ мигрирует в состав **первичного ключа** дочерней сущности.
   - **Неидентифицирующая связь** — дочерняя сущность остаётся независимой. Внешний ключ не входит в состав первичного ключа дочерней сущности.

2. **По обязательности (стр. 85–86):**
   - **Обязательная связь** — при вводе нового экземпляра в дочернюю сущность заполнение атрибутов внешнего ключа обязательно.
   - **Необязательная связь** — обозначается ромбиком.

3. **По мощности (кардинальности) (стр. 86, Таблица 3):**
   - 1 (один)
   - 0..∞ (ноль или много)
   - 0 или 1
   - 1..∞ (один или много)
   - `<число>` (ровно указанное число)

**Наиболее характерные типы связей между сущностями (стр. 86):**

- Связи типа **«часть–целое»** (например, «состоит из», «включает»).
- **Классифицирующие связи** (например, «тип – подтип», «множество – элемент», «общее – частное»).
- **Производственные связи** (например, «начальник–подчинённый»).
- **Функциональные связи** (определяемые глаголами «производит», «влияет», «зависит от», «вычисляется по»).

**Параметры связи (стр. 86):**
- **Имя** — указывается в виде глагола, определяет семантику (смысл) связи.
- **Мощность (кратность, кардинальность)**: 1:1, 1:N, N:M.
- **Тип**: идентифицирующая / неидентифицирующая.
- **Обязательность**: обязательная / необязательная.

**Нотации IDEF1X (стр. 86–88):**

**1. Стандартная нотация (стр. 84–87, Рис. 17, Таблица 3):**
- Прямоугольник с прямыми углами — независимая сущность.
- Прямоугольник с закруглёнными углами — зависимая сущность.
- Сплошная линия — идентифицирующая связь.
- Пунктирная линия — неидентифицирующая связь.
- Точка на конце линии — обозначение связи.
- Рядом указывается мощность.

**2. Нотация «воронья лапка» (Crow’s feet) (стр. 87, Рис. 18, Рис. 19):**  
Широкое распространение получила нотация, где мощность связи отображается специальными символами на конце линии (развилка — «много»).

**3. Стандартная реляционная нотация (стр. 87–88, Рис. 20):**  
Отличием является представление связей **линиями со стрелками**, направленными к первичным ключам сущностей (как принято в оригинальной версии реляционной модели), и выделение полей, обязательных к заполнению. Первичный ключ маркируется особым образом (например, подчёркиванием или ключиком).

**Иерархия наследования (категорий) (стр. 88–89, Рис. 21):**

Применяется, когда две и более сущности имеют общие по смыслу атрибуты или общие по смыслу связи. Для каждой категории требуется указать **дискриминатор** — атрибут суперкласса, который показывает, как отличить одну сущность от другой.

*Два типа категорий (стр. 89):*

| Тип категории | Характеристика |
|---------------|----------------|
| **Полная категория** | Одному экземпляру суперкласса обязательно соответствует экземпляр потомка |
| **Неполная категория** | В суперклассе могут существовать экземпляры, не имеющие соответствующих экземпляров в потомках |

**Этапы разработки реляционной модели с использованием IDEF1X (стр. 89):**

1. **Устранение элементов, не отвечающих принятой модели данных:**
   - Устранение связей N:M (путём ввода промежуточной сущности).
   - Устранение связей, имеющих атрибуты (преобразовать в отдельные сущности).
   - Устранение сложных связей (тернарных и выше).
   - Устранение унарных связей (путём ввода копии сущности и преобразования в бинарную).
   - Устранение многозначных атрибутов (путём введения новой сущности и связи 1:N).
   - Устранение избыточных связей.

2. **Проверка модели с помощью правил нормализации.**

3. **Определение требований поддержки целостности данных.**

**Применение (стр. 89):**  
Методология IDEF1X нашла широкое применение в автоматизированных системах разработки приложений (CASE-системах).

### ✅ Итог простыми словами

> **IDEF1X** — это стандарт для рисования схем баз данных (семантического моделирования), основанный на модели «сущность–связь». Он позволяет построить модель, которая соответствует реляционной БД в 3-й нормальной форме.
>
> **Основные элементы**:
> - Сущности (независимые — прямоугольники, зависимые — с круглыми углами).
> - Атрибуты (ключевые — вверху, необязательные — с пометкой O).
> - Связи (сплошная — идентифицирующая, пунктирная — неидентифицирующая).
>
> **Есть разные нотации (способы рисования)**:
> - Стандартная (точки и линии).
> - «Воронья лапка» (развилки на конце — «много»).
> - Реляционная (стрелки к первичным ключам).
>
> **Категории (наследование)** нужны, когда у нескольких сущностей есть общие атрибуты. Дискриминатор — это поле, которое говорит, к какому подтипу относится объект.

## Вопрос 24. СУБД. Функции СУБД (операции СУБД)

### 📚 Подробное раскрытие по материалу пособия

**Определение СУБД (стр. 92):**

> *«Под СУБД понимают совокупность языковых и программных средств, обеспечивающих создание, поддержание (редактирование) и доступ к данным как со стороны пользователей, так и со стороны приложений»*.

**Что предоставляет СУБД (стр. 92):**

- средства поддержки логической модели данных;
- развитый пользовательский интерфейс;
- средства программирования высокого уровня, с помощью которых можно создавать свои собственные приложения – БД;
- средства администрирования, обеспечения целостности, безопасности и секретности информации.

**Основные операции (функции) СУБД (стр. 92):**

Операции СУБД заключаются в **хранении, управлении, обработке и представлении данных**. Их можно условно разбить на следующие классы:

| Класс операций | Что включает |
|----------------|---------------|
| **a) Операции определения структуры данных** | Создание и изменение схемы БД |
| **b) Операции изменения данных (обновление)** | Добавление (вставка), удаление, модификация (редактирование) |
| **c) Операции поиска и доступа к информации** | Выборка (получение данных), поиск (установка курсора на записи по критерию), сортировка, фильтрация |
| **d) Операции поддержки целостности и восстановления данных** | Обеспечение корректности данных, восстановление после сбоев |
| **e) Операции контроля за доступом** | Разграничение прав пользователей |
| **f) Операции поддержки обмена данными** | Взаимодействие с другими системами |
| **g) Операции администрирования** | Управление БД на уровне администратора |

**Операции на низком уровне (стр. 92):**

- управление данными во внешней памяти;
- управление буферами оперативной памяти;
- управление параллельностью и транзакциями;
- ведение журнала транзакций в БД.

**Языки, предоставляемые СУБД (стр. 92–93):**

1. **Язык описания данных (Data Definition Language – DDL)** — высокоуровневый непроцедурный язык **декларативного типа**, предназначенный для описания логической структуры данных.

2. **Язык манипулирования данными (Data Manipulation Language – DML)** — совокупность инструкций, обеспечивающих выполнение основных операций по работе с данными: ввод, удаление, модификацию и выборку данных. Обычно инструкции DML указывают, **какие** данные необходимо извлечь, а не **как** они должны быть извлечены, поэтому его также называют **декларативным языком**.

**Требования к СУБД (стр. 93):**

Каждая СУБД должна удовлетворять следующим требованиям:

- обеспечивать пользователю возможность создавать новые БД и определять их схему (логическую структуру данных); поддерживать разнообразные представления одних и тех же данных;
- позволять «запрашивать» данные и изменять их с помощью языка манипулирования данными; допускать интеграцию и совместное использование данных различными приложениями;
- поддерживать хранение очень больших массивов данных в течение долгого времени, гарантируя их безопасность и целостность;
- контролировать конкурентный многопользовательский доступ к данным, т. е. гарантировать управление параллельным доступом к данным.

**Структурная схема СУБД (стр. 93, Рис. 23):**

СУБД представляет собой сложную систему, состоящую из многих компонентов. Упрощённая структурная схема включает (стр. 93–94):

| Компонент | Назначение |
|-----------|------------|
| **База данных и словарь данных (системный каталог)** | Хранятся на внешних носителях. Словарь хранит метаданные (описание структуры БД) |
| **Операционная система** | Контролирует операции доступа на низком уровне (запись-чтение). При необходимости высокой производительности эти операции может контролировать сама СУБД |
| **Ядро БД** | Контролирует операции доступа на более высоком уровне, используя информацию из словаря данных. Ответственно за поддержку безопасности и целостности БД |
| **Буфер в оперативной памяти** | Выделяется для каждой операции доступа. Необходимость буферизации обусловлена тем, что скорость обмена с внешней памятью значительно ниже скорости обмена с ОП, а объём ОП всегда меньше объёма внешней памяти. Повышает производительность и позволяет предоставить доступ к одним и тем же данным разным пользователям |
| **Интерпретатор DDL** | Обрабатывает инструкции определения/изменения структуры данных и сохраняет результаты в системном каталоге |
| **Интерпретатор DML** | Интерпретирует запросы от пользователей и передаёт их ядру БД. Запросы (через высокоразвитый интерфейс) проходят предварительную предкомпиляцию в инструкции DML |
| **Операции управления и администрирования** | Выполняются непосредственно ядром БД и промежуточной обработки не требуют |

### ✅ Итог простыми словами

> **СУБД** — это программа, которая управляет базами данных. Она позволяет:
> - **Создавать** структуру БД (таблицы, связи) — через язык DDL.
> - **Добавлять, удалять, изменять и искать** данные — через язык DML.
> - **Обеспечивать** безопасность, целостность, многопользовательский доступ.
>
> **Структура СУБД**:
> - **Ядро** — «мозг», который всё обрабатывает.
> - **Словарь данных** — хранит описание структуры БД.
> - **Буфер** — временная память для ускорения работы.
> - **Интерпретаторы DDL и DML** — переводят команды пользователя в действия ядра.

## Вопрос 25. Классификация СУБД

### 📚 Подробное раскрытие по материалу пособия

На стр. 94 (раздел 5.1) перечислены **признаки, по которым можно различать СУБД**.

**Классификация СУБД по четырём основным признакам:**

**1. По типу поддерживаемой модели данных (стр. 94):**

| Тип | Примеры/примечания |
|-----|---------------------|
| Реляционная | Наиболее распространённая |
| Объектная | ООБД |
| Объектно-реляционная | Гибрид |
| Иерархическая | Классическая |
| Сетевая | Классическая |
| Другие | Многомерная, постреляционная |

**2. По количеству поддерживаемых пользователей (стр. 94):**

| Тип | Характеристика |
|-----|----------------|
| **Однопользовательские** | Работает один пользователь |
| **Многопользовательские** | Одновременно работают несколько пользователей (с контролем параллельного доступа) |

**3. По количеству компьютеров, на которых располагается СУБД (стр. 94):**

| Тип | Характеристика |
|-----|----------------|
| **Централизованные** | СУБД располагается на одном компьютере |
| **Распределённые (Distributed DBMS – DDBMS)** | СУБД располагается на нескольких компьютерах |

**4. По назначению (стр. 94):**

| Тип | Характеристика | Примеры |
|-----|----------------|---------|
| **Общецелевые** | Универсальные СУБД для широкого круга задач | Access, MS SQL Server, Oracle, MySQL, PostgreSQL |
| **Специальные** | Для конкретных областей (обслуживание банков, продажа и резервирование билетов и т.д.) | Часть из них образует категорию **OLTP** (on-line transaction processing systems — системы оперативной обработки транзакций) |

### ✅ Итог простыми словами

> СУБД можно разделить по четырём признакам:
> 1. **По модели данных**: реляционные, объектные, иерархические и др.
> 2. **По числу пользователей**: однопользовательские (для одного) и многопользовательские (для многих).
> 3. **По числу компьютеров**: централизованные (на одном ПК) и распределённые (на нескольких серверах).
> 4. **По назначению**: общецелевые (для всего) и специальные (для конкретных задач, например, банковские системы).

## Вопрос 26. Индексы в БД. Структура и назначение индексов. Первичные, вторичные и составные индексы

### 📚 Подробное раскрытие по материалу пособия

На стр. 94–99 (раздел 5.2) подробно описаны индексы.

**Назначение индексов (стр. 94–95):**

> *«Индексы предназначены для ускорения доступа к данным и повышения скорости поиска. Основным их назначением является устранение последовательного или пошагового сканирования записей при поиске нужных данных»*.

**Дополнительные возможности индексов (стр. 95):**

- проверка значений поля на **уникальность**;
- выполнение **сортировки** записей в таблице.

**Важное уточнение (стр. 95):**  
Не следует путать индексы с первичными ключами. Индекс представляет собой много более сложный объект.

**Как работает ускорение с помощью индексов (стр. 95):**

Ускорение достигается за счёт того, что индекс имеет структуру, оптимизированную под поиск, например:
- **сбалансированное двоичное дерево (B-tree)**;
- **упорядоченный массив** (позволяет применять **метод деления отрезка пополам** — двоичный поиск).

Число обращений к индексу при двоичном поиске: **log₂ n**, где n — число записей в индексе.

**Упрощённое представление индекса (стр. 95):**

Индекс можно представить в виде таблицы с двумя столбцами:

| Столбец 1 | Столбец 2 |
|-----------|-----------|
| Упорядоченные значения индексного выражения (или хеш-код) | Местоположение записи (адрес блока данных) |

*Примечание*: это упрощение. Реально индекс реализуется в виде структуры, оптимизированной под поиск (например, B-tree).

**Хранение хеш-кода вместо значений (стр. 95):**

| Подход | Преимущество | Недостаток |
|--------|---------------|-------------|
| Хранение значений | Прямой доступ | Большая длина полей |
| Хранение хеш-кода | Длина фиксирована (4 байта), независима от исходных значений | Необходимость операции свертки (затраты времени), возможность **коллизий** (разные значения дают одинаковый хеш-код) |

**Способы адресации записей (стр. 95):**

- абсолютные или относительные адреса дисковой памяти;
- так как таблицы хранятся в виде блоков данных фиксированного размера, преимущественно используются адреса начала этих блоков.

---

### 5.2.1. Первичные индексы (стр. 96, Рис. 24)

**Определение (стр. 96):**

> *«Первичный индекс создаётся на основе первичного ключа таблицы, т. е. поля, которое определяет физический порядок следования записей в таблице и является уникальным»*.

**Ограничение (стр. 96):**  
Таблица может содержать не более одного поля, определяющего физический порядок записей, поэтому для неё может быть определён **только один первичный индекс**.

**Структура первичного индекса (стр. 96, Рис. 24):**

| Поле 1 | Поле 2 |
|--------|--------|
| Значение первичного ключа для **первой записи в блоке** данных | Указатель на этот блок данных |

*Важно*: в первичном индексе хранятся не все значения индексируемого поля, а только **первые для каждого блока данных**. Число записей в первичном индексе определяется числом блоков данных таблицы.

**Процесс поиска с использованием первичного индекса (стр. 96):**

1. Поиск в индексе (двоичным методом) нужного значения.
2. Получение указателя на блок данных.
3. Чтение блока данных.
4. Поиск в блоке (последовательный просмотр записей внутри блока).

---

### 5.2.2. Вторичные индексы (стр. 97, Рис. 25)

**Определение (стр. 96–97):**

Дополнительно для таблицы можно определить **несколько вторичных индексов**, основанных на **неупорядоченных физически полях** (не на первичном ключе).

**Отличия вторичного индекса от первичного (стр. 97):**

| Характеристика | Первичный индекс | Вторичный индекс |
|----------------|------------------|------------------|
| Основание | Первичный ключ (уникальный) | Любое поле (может иметь повторяющиеся значения) |
| Физический порядок записей | Определяет | Не определяет |
| Количество на таблицу | Только один | Несколько |
| Структура (поле 2) | Указатель на блок данных | Для повторяющихся значений — список указателей на **все** блоки, содержащие записи с данным значением |

**Рис. 25** (стр. 97) иллюстрирует структуру вторичного индекса, где для упрощения применены указатели на записи (а не на блоки).

---

### 5.2.3. Составные индексы (стр. 97–98)

**Когда применяются (стр. 97):**

Если в таблице поиск часто ведётся **по нескольким полям одновременно**, то для ускорения необходимо определить **составной индекс**.

**Структура составного индекса (стр. 97–98):**

Отличие от обычных индексов: первое поле индекса содержит **свёртку (конкатенацию) значений всех полей**, входящих в индекс.

**Процесс создания составного индекса (стр. 97):**

1. Сортировка по всем полям, входящим в индекс, в порядке их следования (например: Страна → Город → Название предприятия).
2. На основе отсортированных значений создаётся первое поле индекса.
3. Второе поле заполняется указателями на блоки данных.

**Важное замечание (стр. 97–98):**

Совокупность вторичных индексов, созданных для каждого из полей по отдельности, **не сможет заменить составной индекс**, поскольку результат вложенной сортировки не равен результату последовательной сортировки.

**Возможности поиска по составному индексу (стр. 98):**

- Можно организовать поиск по **первому полю**, входящему в составной индекс.
- **Ограничение**: только **последнее условие** (сравнение с последним полем индекса) может быть **неравенством**.

*Пример из пособия (стр. 98):*
```
Страна = 'Беларусь' AND Город = 'Минск' AND Предприятие < 'С'
```
Здесь последнее условие (Предприятие < 'С') — неравенство, что допустимо.

### 5.2.4. Хеш-индексирование (стр. 98–99)

**Определение хеширования (стр. 98):**

> *«Хеширование — преобразование по определённому алгоритму входного массива данных произвольной длины в выходную строку (битовую строку) фиксированной длины»*.

**Определение хеш-индексирования (стр. 98):**

> *«Хеш-индексированием в БД называется технология быстрого прямого доступа к хранимой записи на основе расположения данной записи по адресу, являющемуся результатом вычисления хеш-функции от значения некоторого поля, не обязательно ключевого»*.

**Алгоритм хеширования (стр. 98):**

| Действие | Описание |
|----------|----------|
| **При сохранении записи** | Вычисляется хеш-адрес (хеш-функция от значения хеш-поля), запись помещается по этому адресу |
| **При извлечении записи** | По заданному значению хеш-поля вычисляется хеш-адрес, затем диспетчер файлов извлекает запись по этому адресу |

**Требования к хеш-функции h(K) (стр. 98):**

1. Обеспечивать **равномерное распределение** записей в памяти (для близких значений ключа адреса должны сильно отличаться).
2. Для разных значений ключа выдавать **разные адреса** (на практике допускаются совпадения — коллизии).

**Пример хеш-функции (стр. 98):**

`h(K) = K mod M`, где K — значение хеш-поля, M — количество записей.

**Достоинства хеш-индексирования (стр. 98–99):**

- Обращение к данным по значению ключа происходит **за одну операцию ввода/вывода**.
- **Не нужно создавать дополнительные структуры** типа индекса и тратить память на их хранение.

**Недостатки хеш-индексирования (стр. 99):**

| Недостаток | Пояснение |
|------------|-----------|
| Количество записей и распределение значений должны быть известны заранее | Трудно применять для динамически растущих БД |
| Записи не упорядочены по значению хеш-поля | Дополнительные затраты при сортировке |
| Сложность подбора хеш-функции | Количество значений хеш-поля может значительно превышать количество адресуемых страниц данных |
| Возможность **коллизий** | Две или более различных записей имеют одинаковые адреса. С увеличением количества записей коллизии учащаются, время поиска растёт |

### ✅ Итог простыми словами

> **Индекс** — это как «оглавление» в книге: позволяет быстро найти нужную запись, не перелистывая всё подряд.
>
> **Первичный индекс** — создаётся по первичному ключу (уникальному полю). Он определяет физический порядок записей. Таблица может иметь только один первичный индекс.
>
> **Вторичный индекс** — создаётся по любому полю (даже с повторяющимися значениями). Может быть много. Не определяет физический порядок.
>
> **Составной индекс** — для поиска сразу по нескольким полям (например, Страна + Город + Название).
>
> **Хеш-индексирование** — вычисляет адрес записи по её значению (как по ключу получить номер ячейки). Очень быстро, но нужно заранее знать количество записей, возможны коллизии (когда разные данные попадают в одну ячейку).
