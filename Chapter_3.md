## Конспект главы 3: Модели данных

### Классификация моделей данных

В пособии рассматриваются следующие модели:

| Категория | Модели |
|---|---|
| **Классические (инфологические)** | Иерархическая, сетевая, реляционная |
| **Более поздние** | Постреляционная, многомерная, объектно-ориентированная, объектно-реляционная |
| **Современные** | NoSQL (ключ-значение, документо-ориентированные, колоночные, графовые) |

---

## 3.1. Иерархическая модель

### Структура
- Данные представляются в виде **дерева** (связный граф без циклов).
- **Корень** — вершина, не имеющая родителей.
- **Листья** — вершины, не имеющие потомков.
- **Близнецы** — потомки одного и того же типа.
- Связи: только **1:1** и **1:M**. Связь **M:N** невозможна.

### Пример из учебника
Расписание занятий на факультете: корень — «Факультет», далее — «Курсы», «Группы», «Студенты». Проблема: для каждой группы дублируются записи расписания общих курсов.

### Достоинства
- Эффективность реализации.
- Экономия памяти.
- Хорошее время выполнения основных операций.

### Недостатки (важно для зачёта)
1. **Невозможность реализации связи M:N**.
2. **Сильная логическая зависимость данных** — добавление/удаление вершины без разрушения структуры невозможно.
3. **Избыточность информации** (дублирование).
4. **Затруднённый поиск «снизу вверх»** — приходится перебирать всё.

### Историческая справка
- Первая коммерческая СУБД — **IMS (IBM)** для System/360 (1960-е гг.).
- Языки: ПЛ/1, COBOL.

---

## 3.2. Сетевая модель

### Структура
- Данные — в виде **произвольного графа** (обобщение иерархической модели).
- У потомка может быть **более одного родителя**.
- Записи и связи — основные элементы.

### Достоинства
- **Уменьшение дублирования** информации.
- **Большая гибкость** — можно добавлять новые связи в процессе работы.
- **Симметричные запросы** (сверху вниз и снизу вверх) по схожим алгоритмам.
- Реализация связи **M:N** через записи пересечения.

### Недостатки
1. **Недостаточно высокая скорость поиска**.
2. **Избыточность ссылок** (дублируются ссылки на подчинённые и родительские элементы).
3. **Слабая логическая независимость** (схема всё ещё жёсткая).
4. **Ослабленный контроль целостности** из-за произвольных связей.

### Стандарт
- **CODASYL** — стандарт сетевых БД.
- Примеры СУБД: IDMS, VAX_DBMS, db_VistaIII, «СЕТЬ».

## 3.3. Реляционная модель (РМД) — самая важная

### Основная идея
> Все данные представляются в виде **таблиц (отношений)**, а операции над данными сводятся к операциям над таблицами.

### Четыре свойства таблицы как отношения (строгое определение)

| Свойство | Пояснение |
|---|---|
| 1. Атомарность ячеек | В каждой ячейке — одно скалярное значение, не массив и не составной объект |
| 2. Однородность столбцов | Все значения в столбце — из одного домена |
| 3. Уникальность строк | Нет двух одинаковых строк (кортежей) |
| 4. Произвольный порядок | Строки и столбцы не упорядочены (в отличие от таблицы на экране) |

### Терминология реляционной модели (обязательно запомнить)

| Термин | Математическое определение | Аналог в таблице |
|---|---|---|
| **Отношение (Relation)** | Подмножество декартова произведения доменов | Таблица |
| **Атрибут** | Именованное вхождение домена | Столбец |
| **Домен** | Множество допустимых значений | Тип данных с ограничениями |
| **Кортеж** | Упорядоченный набор значений (d1,...,dn) | Строка |
| **Степень отношения** | Количество атрибутов | Количество столбцов |
| **Кардинальное число** | Количество кортежей | Количество строк |
| **Первичный ключ** | Атрибут(ы), уникально идентифицирующий кортеж | — |
| **Внешний ключ** | Атрибут, ссылающийся на первичный ключ другой таблицы | — |

### Математическое определение (для углублённого понимания)
> Отношение R степени n — это подмножество декартова произведения доменов D₁ × D₂ × ... × Dₙ.

*Пример:* D₁ = {Иванов, Петров}, D₂ = {Физика, Химия}, D₃ = {3,4,5}. Декартово произведение — 2×2×3=18 кортежей. Отношение R может содержать от 0 до 18 кортежей.

### Ключи в реляционной модели

| Термин | Определение |
|---|---|
| **Потенциальный ключ** | Подмножество атрибутов, обладающее **уникальностью** (нет одинаковых кортежей по этим атрибутам) и **неизбыточностью** (ни одно подмножество не обладает уникальностью) |
| **Первичный ключ** | Один из потенциальных ключей, выбранный в качестве основного |
| **Внешний ключ** | Атрибут(ы) подчинённого отношения, значения которого всегда совпадают со значениями первичного ключа базового отношения (или NULL) |

### Важное замечание
> Отношение и таблица — не одно и то же. Отношение — математический объект (строки и столбцы не упорядочены, нет повторов). Таблица — конкретное визуальное представление (строки упорядочены, могут быть повторы).

### Реляционная алгебра (8 основных операций)

#### Теоретико-множественные операторы

| Операция | Обозначение | Результат |
|---|---|---|
| **Объединение** (Union) | A ∪ B | Кортежи, принадлежащие A или B |
| **Пересечение** (Intersect) | A ∩ B | Кортежи, принадлежащие и A, и B |
| **Вычитание** (Minus) | A − B | Кортежи, принадлежащие A, но не B |
| **Декартово произведение** (Cartesian Product) | A × B | Все возможные пары кортежей из A и B |

#### Специальные реляционные операторы

| Операция | Смысл | Аналог в SQL |
|---|---|---|
| **Проекция** (Projection) | Выбор только указанных столбцов | `SELECT DISTINCT столбцы` |
| **Выборка** (Selection) | Выбор строк по условию | `WHERE` |
| **Соединение** (Join) | Композиция двух отношений по равенству значений | `JOIN ... ON` |
| **Деление** (Division) | Кортежи из A, связанные со всеми кортежами из B | (редко используется) |

### Три разновидности соединения

| Тип | Формула | Особенность |
|---|---|---|
| **Θ-соединение** | A × B WHERE X θ Y | Общий случай (θ =, ≠, >, <, ≥, ≤) |
| **Эквисоединение** | A [X = Y] B | Частный случай с равенством |
| **Естественное соединение** | A JOIN B | По всем атрибутам с одинаковыми именами, дублирующий столбец удаляется |

### Свойство замкнутости
> Результат любой реляционной операции над отношениями **сам является отношением**. Это позволяет строить вложенные выражения.

### Запросы, невыразимые реляционной алгеброй
- Транзитивное замыкание (например, все подчинённые сотрудники).
- Кросс-таблицы.
- Некоторые агрегации.

### Связи в реляционной БД

| Тип связи | Как определяется | Пример |
|---|---|---|
| **1:1** | Ключ с обеих сторон | Персональные сведения ↔ Служебные сведения (табельный номер) |
| **1:M** | Ключ с одной стороны, неключевое поле — с другой | Клиенты ↔ Заказы |
| **M:N** | Через промежуточную таблицу с двумя внешними ключами | Клиенты ↔ Товары (через таблицу «Заказы») |
| **Унарная** | Внешний ключ ссылается на первичный ключ той же таблицы | Сотрудник ↔ Начальник (поле «Подчиняется») |

### Целостность в реляционной БД (4 вида ограничений)

| Ограничение | Смысл |
|---|---|
| **Доменные** (domain) | Значение атрибута должно принадлежать домену (тип, маска, диапазон, список) |
| **Ключей** (key) | Уникальность первичных и потенциальных ключей (через уникальные индексы) |
| **Записи** (entity) | Первичный ключ не может быть NULL |
| **Ссылочной целостности** (referential integrity) | Внешний ключ должен ссылаться на существующий первичный ключ (или быть NULL) |

### Два правила внешних ключей

| Правило | Поведение |
|---|---|
| **Ограничение обновления/удаления** | Нельзя удалить запись из базовой таблицы, если есть связанные записи в подчинённой («Master has detail records»). Нельзя вставить внешний ключ с несуществующим значением («Key violation») |
| **Каскадное обновление/удаление** | При изменении первичного ключа — автоматически обновляются внешние ключи. При удалении из базовой таблицы — удаляются все связанные записи из подчинённых |

## 3.4. Нормализация данных (ключевая тема)

### Цели нормализации
1. Исключить ненужное повторение данных.
2. Обеспечить быстрый доступ.
3. Обеспечить целостность (устранить аномалии).

### Три вида аномалий

| Аномалия | Описание | Пример |
|---|---|---|
| **Добавления** | Нельзя добавить данные, пока они не полные | Нельзя добавить новый товар, пока нет покупателя |
| **Удаления** | При удалении теряется не связанная напрямую информация | Удалили последнюю продажу — потеряли описание товара |
| **Модификации** | Изменение одного поля требует просмотра всех записей | Изменили адрес клиента — нужно менять во всех его заказах |

### Нормальные формы (сводная таблица)

| НФ | Дополнительное требование | Что устраняет |
|---|---|---|
| **1НФ** | Атомарные атрибуты + уникальные строки | Составные и многозначные атрибуты |
| **2НФ** | 1НФ + полная ФЗ от первичного ключа | Частичные функциональные зависимости |
| **3НФ** | 2НФ + отсутствие транзитивных зависимостей | Транзитивные зависимости |
| **BCNF** | 3НФ + детерминанты — ключи | Зависимость части составного ключа от неключевого |
| **4НФ** | BCNF + отсутствие нетривиальных многозначных зависимостей | Многозначные зависимости |
| **5НФ** | 4НФ + отсутствие зависимостей объединения | Зависимости объединения (проективно-соединительные) |

### Подробное описание каждой нормальной формы

#### Первая нормальная форма (1НФ)
**Требования:**
1. Нет повторяющихся строк (вводим первичный ключ).
2. Все атрибуты — атомарные (скалярные).

**Действия:**
- Составные поля разложить на простые («Адрес» → «Страна» + «Город» + «Улица»).
- Многозначные поля вынести в отдельные таблицы («Телефоны» отдельно).

**Рекомендация автора:** вводить **составной ключ** на этапе 1НФ, иначе 2НФ будет выполняться автоматически.

#### Вторая нормальная форма (2НФ)
**Требования:**
1. Находиться в 1НФ.
2. Каждый неключевой атрибут должен иметь **полную функциональную зависимость** от первичного ключа (если ключ составной — не может зависеть от части).

**Что делаем:** выносим атрибуты, зависящие от части ключа, в отдельные таблицы.

*Пример:* Продажи с ключом (Код_клиента, Код_товара, Дата_заказа). Атрибуты клиента зависят только от Код_клиента → выносим в таблицу «Клиенты». Атрибуты товара зависят только от Код_товара → выносим в таблицу «Товары».

#### Третья нормальная форма (3НФ)
**Требования:**
1. Находиться в 2НФ.
2. Нет транзитивных зависимостей (неключевой атрибут не должен зависеть от другого неключевого).

*Пример:* В таблице «Товары» поле «Размещение» зависит от поля «Категория» (не от ключа). Выносим «Категории» в отдельную таблицу.

#### Нормальная форма Бойса — Кодда (BCNF)
**Требования:**
1. Находиться в 3НФ.
2. Каждый детерминант — потенциальный ключ.

**Когда нужна:** когда есть зависимость **части составного ключа** от неключевого атрибута.

*Пример:* Студент(Номер_студента, Факультет, Декан). Декан зависит от факультета, а факультет — часть составного ключа? Требует декомпозиции.

**Важно:** Если у отношения единственный потенциальный ключ, 3НФ и BCNF эквивалентны.

#### Четвёртая нормальная форма (4НФ)
**Требования:**
1. Находиться в BCNF.
2. Нет **нетривиальных многозначных зависимостей**.

**Многозначная зависимость (MVD):** X →→ Y (одному X соответствует множество Y).

**Тривиальная MVD:** Y ⊂ X или Y ∪ X = R (все атрибуты). Не нарушает 4НФ.

**Нетривиальная MVD:** требует дальнейшей декомпозиции (часто сразу на три таблицы).

*Пример:* Заказы: Код_заказа →→ Сумма_оплаты и Код_заказа →→ Вид_оплаты. Создаём таблицы «Оплата» и «Виды_оплаты».

#### Пятая нормальная форма (5НФ, проективно-соединительная)
**Требования:** отсутствие **зависимостей объединения**.

**Суть:** бывают ситуации, когда таблицу нельзя разбить на две без потерь, но можно разбить на три и более. 5НФ устраняет такие аномалии.

### Денормализация

**Определение:** намеренное приведение структуры БД в состояние, не соответствующее нормальным формам, для **ускорения чтения**.

| Параметр | Нормализация | Денормализация |
|---|---|---|
| Цель | Уменьшение избыточности, целостность | Скорость запросов SELECT |
| Плата | Медленные JOIN-ы | Риск аномалий при модификации |
| Когда применять | Всегда по умолчанию | Когда JOIN более 5-6 таблиц и ускорение иначе невозможно |

**Виды денормализации:**
1. **Восходящая** — перенос данных из подчинённой таблицы в родительскую.
2. **Нисходящая** — перенос из родительской в подчинённую.
3. **Разбиение** — большие редкоиспользуемые поля (BLOB) выносятся в отдельную таблицу, а для запросов создаётся представление (VIEW).

### Правила нормализации (5 правил из учебника)

1. Разрабатывайте схему так, чтобы её можно было легко объяснить — не комбинируйте атрибуты независимых объектов, не создавайте сложные связи.
2. Исключайте возможность появления аномалий обновления.
3. В связях участвуют только первичные (потенциальные) и внешние ключи — это предотвращает появление «подложных» записей.

**Важное замечание:** требование, чтобы все таблицы были нормализованы, — **необязательное**. В некоторых случаях можно оставить ненормализованные таблицы (например, «Телефоны» можно и не выносить, если телефонов мало и они редко меняются).

### Выводы по реляционной модели

**Основные свойства РМД:**
1. Основная единица обработки — отношение (результат выборки — тоже отношение).
2. Данные представляются только явными скалярными значениями (никаких указателей).
3. Нет указателей на строки — идентификация только по ключу.

**Назначение ключей:**
1. Исключение дублирования кортежей и их адресация.
2. Организация связывания отношений.

**Достоинства:**
- Простота и наглядность.
- Гибкость.
- Точность (математическая основа).
- Логическая независимость данных.
- Ясная картина связей.

**Недостатки:**
- Однородная структура данных (сложные объекты — во многие таблицы).
- Возможность повторного присвоения идентификатора (теряется идентифицируемость).
- Запрет многозначных и составных полей.
- Сложность описания рекурсивных, иерархических, сетевых связей.
- Фиксированный набор операций (нельзя определить новые).

**Примеры реляционных СУБД:** dBase, Paradox, Access, DB2, MS SQL Server, InterBase, Oracle, MySQL, Firebird, PostgreSQL, SQLite.

**Классификация по архитектуре:**
- **Встраиваемые (embedded):** SQLite, Local DB.
- **Локальные:** Paradox, Access, Adabas D.
- **Клиент-серверные:** MySQL, Firebird, PostgreSQL, Oracle, DB2, MS SQL Server, Informix, Sybase.

## 3.5. Постреляционная модель (Non-First Normal Form, NF²)

### Основная идея
Снимает ограничение 1НФ — допускает **многозначные и составные поля**. Вложенные таблицы (nested tables) как значения атрибутов.

### Обозначение в таблице (пример из учебника)
Таблица с полем «Телефоны», содержащим несколько строк с разными типами телефонов (дом., раб., моб.).

### Достоинства
- Более эффективное хранение данных.
- Не требуется выполнять JOIN для получения вложенной информации.
- Высокая наглядность.

### Недостатки
- Сложность обеспечения целостности.
- Проблема: как проверить противоречивость внутри вложенной таблицы?

### Реализации
- **Oracle 8+** — типы OBJECT (составные) и VARRAY (массивы переменной длины), вложенные таблицы (Nested Tables).
- **PostgreSQL** — составные типы, массивы (включая многомерные), типы-перечисления.

### Пример кода Oracle
```sql
CREATE TYPE Tel_Type AS OBJECT (Tel_Number CHAR(10), Tel_Descr CHAR(5));
CREATE TYPE Tel_List_Type AS VARRAY(5) OF Tel_Type;
CREATE TYPE Address_Type AS OBJECT (street VARCHAR2(100), home NUMBER(2), appartment NUMBER(2));
CREATE TABLE clients (client_id NUMBER(6), first_name VARCHAR2(20), phone_number Tel_List_Type, address Address_Type);
```

## 3.6. NoSQL базы данных

### Расшифровка и суть
**NoSQL** = «Not Only SQL» (не только SQL). Это не конкретная модель, а семейство подходов.

### Причины появления
1. **Масштабируемость** — реляционные БД плохо масштабируются горизонтально (на много серверов). Когда ресурсы одного сервера кончаются, нужны сотни серверов — сложность растёт взрывным образом.
2. **Скорость JOIN** — в больших распределённых системах операция соединения таблиц становится узким местом.
3. **Облачные сервисы** требуют масштабируемых хранилищ.

### CAP-теорема (упоминается в тексте)
В распределённой системе можно обеспечить только два из трёх свойств:
- **Consistency** (согласованность)
- **Availability** (доступность)
- **Partition tolerance** (устойчивость к разделению)

### BASE вместо ACID

| ACID (реляционные) | BASE (NoSQL) |
|---|---|
| Atomicity (атомарность) | Basic Availability (базовая доступность — каждый запрос завершается) |
| Consistency (согласованность) | Soft state (гибкое состояние — может меняться без ввода) |
| Isolation (изолированность) | Eventual consistency (согласованность в конечном счёте) |
| Durability (надёжность) | — |

**Eventual consistency** — данные могут быть некоторое время рассогласованы, но в итоге придут к согласованному состоянию.

### Типы NoSQL хранилищ

| Тип | Суть | Примеры |
|---|---|---|
| **Ключ-значение** (Key-Value) | Простейшее хранилище, доступ по ключу | Berkeley DB, MemcacheDB, Redis, Amazon DynamoDB |
| **Документо-ориентированные** | JSON-подобные документы, коллекции | CouchDB, MongoDB |
| **Семейства колонок** (Column family) | Разреженная матрица, строка+колонка+время | Apache HBase, Apache Cassandra, SimpleDB |
| **Графовые** | Упор на связи между объектами | OrientDB |

### Признаки NoSQL (из учебника)
1. Не используется SQL (или не только SQL).
2. **Schemaless** — слабо типизированная структура (можно добавить поле динамически).
3. Данные — в виде **агрегатов** (целостных объектов, минимум JOIN).
4. Слабые ACID-свойства или BASE.
5. Возможность распределённых систем **shared nothing** (без совместных ресурсов).

### Проблема schemaless
Добавили новое поле в одном документе — его нет в остальных. Два решения:
1. Обойти все документы и добавить (накладные расходы).
2. Проверять наличие поля в коде приложения (замедление).

### Когда NoSQL не вытеснит реляционные
- Реляционные БД обладают гораздо более высокой безопасностью и согласованностью.
- Проблемы масштабируемости возникают на десятках и сотнях терабайт данных — такие объёмы есть только у сотен компаний в мире.

## 3.7. Многомерная модель

### Назначение
**OLAP** (On-Line Analytical Processing) — оперативная аналитическая обработка.  
**Data Warehousing** — хранилища данных.  
**Data Mining** — интеллектуальный анализ данных (поиск новых знаний: предсказание, идентификация, классификация, снижение размерности, оптимизация).

### Четыре требования к OLAP-системе (FAST — Shared, Multidimensional, Information?)
- **Fast** — быстрый отклик.
- **Shared** — многопользовательская (контроль доступа).
- **Multidimensional** — многомерное представление.
- **Information** — информативность (полнота данных).

### Базовые понятия многомерной модели

| Понятие | Определение |
|---|---|
| **Измерение (Dimension)** | Множество однотипных данных, образующих грань гиперкуба (аналог ключа) |
| **Ячейка (Cell)** | Показатель, значение которого определяется координатами по измерениям |
| **Гиперкуб** | Многомерное представление данных (куб, если 3 измерения) |

### Операции над гиперкубом

| Операция | Смысл |
|---|---|
| **Консолидация (Aggregation)** | Расчёт итоговых значений (сумма, среднее, ...) |
| **Срез (Slice)** | Фиксация одного или нескольких измерений (подмножество куба) |
| **Вращение (Rotate)** | Изменение порядка измерений при визуализации |
| **Детализация (Drill Down)** | Переход от общего к более детальному представлению |
| **Обобщение (Drill Up)** | Переход к более общему представлению |

### Классификация агрегирующих функций

| Тип | Описание | Пример |
|---|---|---|
| **Дистрибутивные** | Можно разбить на части, вычислить отдельно, потом объединить | SUM, COUNT |
| **Алгебраические** | Комбинация дистрибутивных | AVG (SUM/COUNT) |
| **Холистические** | Нельзя вычислить на частичных данных | MEDIAN, MODE |

### Способы создания OLAP-кубов

| Тип | Хранение данных | Плюсы | Минусы |
|---|---|---|---|
| **ROLAP** | В реляционных таблицах (все агрегации) | Единый формат, одна СУБД | Рост объёма, сложность пересчёта |
| **MOLAP** | В многомерной БД | Высокая скорость | Отрыв от базовых данных, спец. инструменты |
| **HOLAP** | Базовые — реляционно, агрегаты — многомерно | Комбинация плюсов | — |

### Схемы хранения в реляционной БД

| Схема | Описание |
|---|---|
| **Звезда (Star)** | Одна денормализованная таблица фактов + несколько таблиц измерений (связь 1:M). Таблица фактов содержит числовые поля для агрегации |
| **Снежинка (Snowflake)** | Расширение звезды для иерархических измерений (несколько таблиц фактов для разных уровней обобщения) |

### Средства работы с многомерными данными
- Microsoft Excel Pivot Tables.
- Crystal Reports.
- Cubes (Python framework + HTTP-сервер + Cubesviewer).
- Расширения SQL: **GROUP BY CUBE, GROUP BY ROLLUP, GROUPING SET, WINDOW BY, MODEL BY** (в Oracle 10g+).

### Достоинства многомерной модели
- Удобство и эффективность аналитической обработки больших объёмов (особенно временных рядов).
- Автоматический расчёт итогов по всем размерностям и их сочетаниям (Grand Total).

### Недостатки многомерной модели
- Громоздкость для простых задач оперативной обработки.
- При использовании реляционной модели — нелинейный рост трудоёмкости и затрат памяти на индексацию.

## 3.8. Объектно-ориентированная модель (ООМД)

### Назначение
Длительное хранение объектов программ, чтобы другие программы могли их использовать (в отличие от файлов, где каждая программа пишет свой конвертер).

### Проблема, которую решает
**Семантический разрыв (impedance mismatch)** между объектно-ориентированным кодом и реляционной БД. Постоянное преобразование «объекты ↔ таблицы»:
- Занимает до 30% времени выполнения.
- Требует написания конвертеров.
- ORM (Object-Relational Mapping: Hibernate, NHibernate) автоматизирует, но всё равно медленнее ручного кода.

### Ключевые понятия ООМД

| Понятие | Определение |
|---|---|
| **Класс** | Определённый пользователем инкапсулированный тип объектов произвольной сложности (шаблон) |
| **Объект (экземпляр)** | Конкретный представитель класса с собственными значениями атрибутов |
| **Атрибут** | Характеристика объекта (данные) |
| **Метод** | Операция над атрибутами (состоит из подписи/интерфейса и тела/имплементации) |
| **Инкапсуляция** | Данные и методы внутри объекта; доступ к данным — только через методы объекта |
| **Наследование** | Класс может быть определён на основе другого (суперкласс → подкласс). Обобщение (суперкласс) и специализация (подкласс) |
| **Полиморфизм** | Один и тот же код работает с разными типами данных; позднее связывание |

### Идентификация объектов — OID (Object Identifier)

**Свойства OID:**
1. Генерируется СУБД автоматически.
2. Уникально идентифицирует объект, не зависит от значений атрибутов.
3. **Инвариантен** — не меняется в течение жизни объекта, не может быть использован повторно (даже после удаления).

**Преимущества OID:**
- Эффективность (фактически указатель на адрес в памяти).
- Независимость от данных (объект может менять содержание, оставаясь тем же объектом).
- Пользователь не может изменить.

**Недостаток:** возможность создания двух идентичных копий объекта с разными OID — поэтому ключевые атрибуты всё равно нужны.

### Доступность и перманентность объектов

| Механизм | Суть | Недостатки |
|---|---|---|
| **Контрольные точки** | Копирование всего адресного пространства программы | Только для создавшей программы, много мусора |
| **Сериализация** | Обход графа объектов, запись на диск | Проблема совместного использования подчинённых структур, большие блоки |
| **Явный обмен страницами** | Отображение указателей оперативной ↔ внешней памяти | Сборка мусора, работа с двумя системами указателей |

### Стандарт ODMG (Object Database Management Group)

**Состав стандарта ODMG-2:**
1. Объектная модель.
2. Язык определения объектов (ODL).
3. Объектный язык запросов (OQL) — похож на SQL + наследование + инкапсуляция + полиморфизм.
4. Расширения для C++, Java, Smalltalk.

**Ключевые слова:** `attribute`, `key`, `relationship`, `extent`.

### Достоинства ООМД
1. Однозначная идентификация объектов (OID).
2. Расширенные возможности моделирования (ближе к реальному миру).
3. Расширяемость (можно добавлять новые классы, не ломая старые).
4. Устранение проблемы несоответствия (нет конвертации объект ↔ таблица).
5. Поддержка эволюции схемы.
6. Применимость для сложных специализированных БД.

### Недостатки ООМД
1. Отсутствие универсальной модели данных.
2. Недостаточный опыт эксплуатации, отсутствие стандартов (кроме ODMG).
3. Сложность.
4. Отсутствие поддержки стандартных видов представлений.
5. Недостаточность средств целостности и безопасности.

### Концептуальные различия ООМД и РМД

| Аспект | Реляционная модель | Объектно-ориентированная модель |
|---|---|---|
| Типы данных | Простые (домены) | Классы (пользовательские) |
| Идентификация | Потенциальные ключи | OID (автоматический) |
| Связи | Первичные и внешние ключи | Инверсные ссылки (OID) |
| Наследование | Нет | Есть |
| Операции | Общедоступные, после структуры | Принадлежат объектам, вместе с ними |

**Примеры ООБД:** OpenODB, Iris (HP), O2 (Ardent Software), Object Store (Object Design).

## 3.9. Объектно-реляционная модель (ОРМД)

### Основная идея
> Домен (реляционная модель) и класс (объектно-ориентированная) — концептуально одно и то же. Если разрешить доменам быть классами произвольной сложности с методами, то реляционная модель сможет делать всё то же, что и ООБД.

### Объединяет
- **Реляционный подход:** отношения содержат сами объекты (концептуально), а не ссылки или указатели.
- **Объектно-ориентированный подход:** пользовательские типы данных произвольной сложности + методы.

### Результат
Реляционная модель, где в качестве доменов могут выступать классы. Таблицы могут содержать объекты целиком.

## Ответы на контрольные вопросы главы 3 (31 вопрос)

### 1. Основные модели данных.
Иерархическая, сетевая, реляционная, постреляционная, многомерная, объектно-ориентированная, объектно-реляционная, NoSQL.

### 2. Достоинства и недостатки иерархической модели.

| Достоинства | Недостатки |
|---|---|
| Эффективность, экономия памяти, хорошее время операций | Невозможность M:N, сильная логическая зависимость, избыточность, затруднённый поиск снизу |

### 3. Достоинства и недостатки сетевой модели.

| Достоинства | Недостатки |
|---|---|
| Уменьшение дублирования, гибкость, симметричные запросы | Низкая скорость поиска, избыточность ссылок, слабая независимость, ослабленная целостность |

### 4. Какой вид связи невозможно реализовать в иерархической модели?
**M:N (многие ко многим)**. Только 1:1 и 1:M.

### 5. Условия, при которых таблицу можно считать отношением.
1. Атомарность ячеек.
2. Однородность столбцов.
3. Уникальность строк.
4. Произвольный порядок строк и столбцов.

### 6. Определения терминов реляционной модели.

| Термин | Определение |
|---|---|
| Отношение | Подмножество декартова произведения доменов |
| Домен | Множество допустимых значений атрибута |
| Кортеж | Упорядоченный набор значений (строка) |
| Степень отношения | Количество атрибутов (столбцов) |
| Кардинальное число | Количество кортежей (строк) |
| Первичный ключ | Атрибут(ы), уникально идентифицирующий кортеж |
| Внешний ключ | Атрибут, ссылающийся на первичный ключ другой таблицы |

### 7. Различия между потенциальными ключами и первичным ключом.
**Потенциальные ключи** — все наборы атрибутов, обладающие уникальностью и неизбыточностью. **Первичный ключ** — один из них, выбранный в качестве основного. Остальные потенциальные ключи называются **альтернативными**.

### 8. Идентификатор записи в РМД.
**Ответ:** **а) значение первичного ключа** (и б) значение потенциального ключа — тоже да, но первичный — основной). Внешний ключ не идентифицирует запись уникально. Порядковый номер — нереляционный подход.

### 9. Какие термины означают одно и то же?

| Синонимы |
|---|
| Отношение = Таблица |
| Кортеж = Строка = Запись |
| Атрибут = Столбец = Поле |
| Домен — отдельно (множество допустимых значений) |

### 10. Виды связей между отношениями на примерах.
- **1:1** — Персональные сведения (Код_пс) ↔ Служебные сведения (Код_сс)
- **1:M** — Клиенты (один) → Заказы (много)
- **M:N** — Клиенты ↔ Товары (через таблицу «Заказы»)

### 11. Средства обеспечения ссылочной целостности.
- Правило **ограничения обновления/удаления** (запрет изменять/удалять базовый ключ, если есть связанные записи; запрет вставлять внешний ключ с несуществующим значением).
- Правило **каскадного обновления/удаления** (автоматическое обновление внешних ключей или удаление подчинённых записей).

### 12. Что даёт каскадное обновление записей?
При изменении значения первичного ключа в базовой таблице все внешние ключи в подчинённых таблицах автоматически обновляются до нового значения.

### 13. Что даёт каскадное удаление записей?
При удалении записи из базовой таблицы автоматически удаляются все связанные записи из подчинённых таблиц.

### 14. Отличительные свойства РМД.
1. Основная единица — отношение.
2. Данные — только явные скалярные значения (никаких указателей).
3. Нет указателей на строки — идентификация по ключу.

### 15. Назначение ключей в РМД.
1. Исключение дублирования кортежей и адресация кортежей.
2. Организация связывания отношений (через внешние ключи).

### 16. Задачи теории нормализации данных.
Исключить ненужное повторение данных, обеспечить быстрый доступ, обеспечить целостность данных (устранить аномалии добавления, удаления, модификации).

### 17. Задачи при выполнении денормализации.
Ускорение операций чтения (SELECT) за счёт добавления избыточных данных (уменьшение количества JOIN). Применяется, когда JOIN 5-6 таблиц выполняется неприемлемо долго.

### 18. Проблемы, связанные с наличием повторяющихся данных.
- Аномалии добавления (нельзя добавить неполные данные).
- Аномалии удаления (при удалении теряется другая информация).
- Аномалии модификации (изменение требует просмотра всех записей).
- Неоправданное увеличение размера.
- Снижение скорости выполнения запросов.

### 19. Правила нормализации (из учебника).
1. Разрабатывайте схему так, чтобы её можно было легко объяснить (не комбинируйте атрибуты независимых объектов, не создавайте сложные связи).
2. Исключайте возможность появления аномалий обновления.
3. В связях участвуют только первичные (потенциальные) и внешние ключи (избегайте «подложных» записей).

### 20. Определения нормальных форм.

| НФ | Определение |
|---|---|
| **1НФ** | Нет повторяющихся строк, все атрибуты атомарные |
| **2НФ** | 1НФ + полная функциональная зависимость неключевых атрибутов от первичного ключа |
| **3НФ** | 2НФ + отсутствие транзитивных зависимостей неключевых атрибутов |
| **BCNF** | 3НФ + каждый детерминант — потенциальный ключ |
| **4НФ** | BCNF + отсутствие нетривиальных многозначных зависимостей |
| **5НФ** | 4НФ + отсутствие зависимостей объединения (проективно-соединительных) |

### 21. Зачем разбивать информацию на несколько связанных таблиц?
Чтобы устранить аномалии обновления, уменьшить избыточность, обеспечить целостность. Каждая таблица должна описывать **один тип объекта**.

### 22. Реляционная модель расписания занятий (Рис. 11).
Должна содержать таблицы: **Группы** (Код_группы), **Преподаватели** (Код_преподавателя), **Дисциплины** (Код_дисциплины), **Аудитории** (Код_аудитории), **Расписание** (внешние ключи на все таблицы + День, Время), и для связи студентов с преподавателями — **Задания** (внешние ключи + описание задания).

### 23. Реляционная модель предприятия (Рис. 4).
Рис. 4 в учебнике — ER-диаграмма. ### 24. Достоинства и недостатки РМД.

| Достоинства | Недостатки |
|---|---|
| Простота, наглядность | Однородная структура (сложные объекты — в много таблиц) |
| Гибкость | Проблема повторного присвоения идентификатора |
| Точность (математическая основа) | Запрет многозначных/составных полей |
| Логическая независимость данных | Сложность рекурсивных/иерархических/сетевых связей |
| Ясная картина связей | Фиксированный набор операций |

### 25. Отличие постреляционной модели от реляционной.
Постреляционная (NF²) **снимает ограничение 1НФ**: допускает многозначные и составные поля, вложенные таблицы. В реляционной всё должно быть атомарным.

### 26. Назначение и принципы ООБД.
**Назначение:** длительное хранение объектов программ, доступных другим программам.

**Принципы:**
- Классы (инкапсулированные типы произвольной сложности).
- Наследование.
- Полиморфизм.
- OID (автоматическая уникальная идентификация).
- Инверсные ссылки для связей.

### 27. Достоинства и недостатки многомерной модели.

| Достоинства | Недостатки |
|---|---|
| Удобство и эффективность аналитической обработки (OLAP) | Громоздкость для простых оперативных задач |
| Автоматический расчёт итогов по всем размерностям | Нелинейный рост трудоёмкости в реляционной реализации |
| Наглядность для аналитиков | — |

### 28. Относительное сходство какого понятия привело к ОРБД?
**Домен (реляционная модель) и класс (объектно-ориентированная).** Если домен может быть классом произвольной сложности с методами, то реляционная модель получает возможности ООБД.

### 29. Принципиальные отличия РМД, ООМД и ОРМД.

| Аспект | РМД | ООМД | ОРМД |
|---|---|---|---|
| Типы данных | Простые домены | Классы (произвольная сложность) | Домены как классы |
| Идентификация | Потенциальные ключи | OID | OID или ключи |
| Связи | Внешние ключи | Инверсные ссылки (OID) | Оба подхода |
| Наследование | Нет | Есть | Есть (через домены-классы) |
| Операции | Общедоступные | Принадлежат объектам | Принадлежат объектам |

### 30. Определение многомерной модели. Отличие от других.
**Определение:** данные представляются в виде многомерного гиперкуба, где измерения — это категории (аналоги ключей), а ячейки — показатели.

**Главное отличие:** ориентация на **аналитическую обработку** (OLAP), автоматический расчёт итогов (консолидация) по всем возможным комбинациям измерений.
