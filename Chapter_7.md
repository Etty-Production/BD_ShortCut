## Конспект главы 7: Архитектура баз данных

### Предпосылки возникновения распределённого доступа

Современные организации имеют географически распределённую структуру, большой штат сотрудников, которым может потребоваться доступ к общим корпоративным данным в любой момент времени. Простое дублирование информации на каждом персональном компьютере не решает проблему, так как данные могут динамически меняться. Хранение больших объёмов данных на каждом компьютере ресурсоёмко. Требуются технологии совместного доступа к данным и системы управления и обработки данных.

---

## 7.1. Технологии доступа к данным Microsoft

### Эволюция технологий

| Технология | Назначение | Недостатки |
|---|---|---|
| **Clipboard** (буфер обмена) | Копирование и вставка данных между приложениями | Ручной режим, отсутствие динамической связи |
| **DDE** (Dynamic Data Exchange) | Протокол динамического обмена данными | Медленный, ненадёжный |
| **OLE** (Object Linking and Embedding) | Создание составных приложений, встраивание/связывание объектов | — |
| **COM** (Component Object Model) | Двоичный стандарт взаимодействия объектов (не зависит от языка) | — |
| **DCOM** (Distributed COM) | COM для распределённых систем (замена межпроцессных сообщений сетевыми протоколами) | — |

---

### OLE (Object Linking and Embedding)

**Назначение:** создание составных приложений, включающих объекты из других приложений.

**Термины:**
- **Контейнер OLE** — приложение, включающее чужие объекты.
- **Сервер OLE** — приложение, поставляющее объекты для встраивания/связывания.

**Типы включения объектов:**

| Тип | Что хранится | Поведение |
|---|---|---|
| **Встраивание (embedding)** | Копия объекта внутри контейнера | Объект редактируется в контейнере (запускается сервер) |
| **Связывание (linking)** | Ссылка на объект-источник | При обновлении исходного файла обновляется представление |

---

### COM (Component Object Model)

**COM** — объектно-ориентированная модель, определяющая:
1. Спецификацию интерфейса между объектами внутри системы.
2. Конкретную реализацию в виде **DLL** (Dynamic Link Library).

**Основное достоинство:** предоставляет **унифицированный (двоичный) стандарт взаимодействия**, не зависящий от языка программирования, на котором созданы приложение и объект.

**Реализация:** в 1993 году в спецификации OLE 2.0.

---

### DCOM (Distributed Component Object Model)

**Назначение:** расширение COM для распределённых корпоративных систем.

**Принцип:** компоненты одинаково выглядят для клиентов на локальном и удалённом компьютерах. Заменяет сообщения между процессами соответствующими сетевыми протоколами.

---

### ODBC (Open Database Connectivity)

**Проблема, которую решает:** каждая фирма-разработчик СУБД использовала свой диалект SQL, что делало невозможным обмен данными между БД разных форматов.

**Решение:**
1. Разработан общий стандарт **CLI** (Call Level Interface) на основе стандартов X/Open и ISO.
2. Каждая фирма-разработчик создала **драйвер перевода** своего диалекта SQL в CLI и обратно.

**Назначение ODBC:** абстрагирование приложения от особенностей ядра используемой БД.

**Архитектура:** дополнительный уровень между приложением и СУБД. Приложение указывает только **DSN** (Data Source Name) — имя источника данных. Функции, драйверы, адреса серверов, сети и шлюзы скрыты от пользователя.

**Достоинства:** простота разработки (высокий уровень абстракции).

**Недостатки:** необходимость трансляции запросов → снижение скорости доступа.

---

### OLE DB (Object Linking and Embedding Database)

**Назначение:** универсальный доступ к любым типам данных (не только реляционным БД), а именно:
- Реляционные БД
- Объектно-ориентированные БД
- Электронные таблицы
- Документы в RTF, XML
- Почтовые системы

**Механизм — провайдеры (providers):**
- **Поставщики данных (data providers)** — надстройка над физическим форматом данных (COM-компонент, принимающий вызовы OLE DB API).
- **Сервис-провайдеры** — реализуют сервисные функции.

**Связь с ODBC:** OLE DB может использовать ODBC для доступа к реляционным БД (через OLE DB-провайдер для ODBC).

> OLE DB не заменяет ODBC, а дополняет её.

---

### DAO и ADO (Data Access Objects / ActiveX Data Objects)

**Проблема:** ODBC и OLE DB — низкоуровневые интерфейсы, имеют много ограничений.

**Решение:** создание высокоуровневых объектных моделей (библиотек функций) — ещё один уровень абстракции.

| Технология | Назначение | Особенность |
|---|---|---|
| **DAO** | Преимущественно для БД Access | Прямой доступ к функциям ядра Microsoft Jet |
| **ADO** | Доступ к любым OLE DB-провайдерам | Ещё более высокий уровень абстракции |
| **ADO.NET** | Современная платформа .NET | Для работы с распределёнными БД по архитектуре клиент-сервер |

**Особенность ADO:** возможность использования в приложениях Intranet/Internet для доступа к различным источникам данных.

---

## 7.2. Архитектура клиент-сервер

### Файл-серверная архитектура (исторически первая)

**Принцип:** по запросам пользователей **файлы БД целиком передаются** на ПК пользователя, где и производится их обработка.

**Особенности:**
- Физические файлы БД на файл-сервере (мощный компьютер).
- Для каждого клиента создаётся локальная копия данных.
- Забота о целостности данных при одновременном доступе целиком ложится на программы клиентов.

**Проблемы:**
- Высокая загрузка сети (даже для выборки нескольких строк передаётся вся таблица).
- При росте числа компьютеров или объёма БД производительность резко падает.
- Нет встроенных механизмов синхронизации параллельного доступа.

---

### Архитектура клиент-сервер (общее определение)

| Компонент | Где находится | Функции |
|---|---|---|
| **Клиент (front-end)** | На компьютерах пользователей | Интерактивный интерфейс, представление данных |
| **Сервер (back-end)** | На выделенном компьютере | Управление данными, обработка запросов, администрирование, целостность, безопасность |

**Достоинства:**
- Сочетание централизованного хранения с индивидуальной работой пользователей.
- Легко реализуется многопользовательский доступ.
- Снижается нагрузка на сеть (передаётся только результат запроса, а не вся таблица).
- Обработка транзакций и бизнес-логики — на сервере.

**Протокол коммуникации:** SQL (стандартный интерфейс для СУБД различных типов). Сервер в таком случае называется **SQL-сервером**.

---

### Типы архитектур по разделению функций (Рис. 31)

| Архитектура | Где выполняется логика | Где выполняется представление | Особенности |
|---|---|---|---|
| **Удалённый доступ к данным (RDA)** | На клиенте | На клиенте | Весь объём данных передаётся на клиент. Прикладные функции тесно связаны с представлением. Достоинство: много готовых инструментов. Недостатки: высокая загрузка сети, сложность модификации. Фактически не отличается от файл-сервера |
| **Сервер БД (DBS)** — удалённое представление | На сервере (хранимые процедуры, триггеры) | На клиенте | Клиент — «тонкий». Логика — на сервере (хранимые процедуры, функции, триггеры). Хранимые команды выполняются быстрее (нет синтаксического разбора и оптимизации при каждом вызове). Достоинства: централизованное администрирование, эффективное использование ресурсов. Недостатки: ограниченность действий через хранимые процедуры |
| **Распределённая функция (Distributed function)** | Распределена между клиентом и сервером | На клиенте | Общая часть логики — на сервере (стандартная целостность), специфическая — на клиенте |
| **Распределённое представление** | На сервере | На сервере | Клиент практически вырожден, только отображение и связь |
| **Распределённая БД** | Распределена | Распределена | Клиент фактически равен серверу, данные могут храниться и на клиенте, и на сервере |

---

### Тонкий vs толстый клиент

| Тип | Характеристика |
|---|---|
| **Тонкий клиент** | Вся обработка данных — на сервере. Клиент — только интерфейс и представление |
| **Толстый клиент** | Часть обработки данных производится на стороне клиента |

---

### Многоуровневая (трёхзвенная) архитектура

```
Уровень 1 (клиент)      → Пользовательский интерфейс (ПК пользователей)
Уровень 2 (сервер приложений) → Бизнес-логика, обработка
Уровень 3 (сервер БД)   → Хранение данных
```

## 7.3. Распределённые базы данных (РБД)

### Определения

| Термин | Определение |
|---|---|
| **Распределённая БД (DDB)** | Совокупность логически взаимосвязанных БД, распределённых в компьютерной сети |
| **Распределённая СУБД (DDBMS)** | Совокупность программ для управления распределённой БД |

**Распределённая вычислительная система** — совокупность элементов (не обязательно однородных), соединённых коммуникационной сетью и взаимодействующих при решении общей задачи.

**Преимущества распределённой системы:**
1. Увеличение мощности при решении общей задачи.
2. Возможность автономной работы отдельных элементов.

---

### Организация распределённого хранения данных

| Способ | Описание |
|---|---|
| **Фрагментация** | Разбиение данных на части (фрагменты), размещённые на разных компьютерах |
| **Репликация (тиражирование)** | Размещение копий данных (одного или нескольких фрагментов) на нескольких компьютерах |

---

#### Фрагментация

| Тип фрагментации | Описание |
|---|---|
| **Горизонтальная** | Разбиение по строкам (непересекающиеся группы строк с одинаковой структурой) |
| **Вертикальная** | Разбиение по столбцам (в каждый фрагмент добавляется первичный ключ для идентификации) |

**Дополнительные термины:**

| Термин | Описание |
|---|---|
| **Секционирование** | Размещение фрагментов в пределах одного физического сервера |
| **Шардинг (sharding)** | Разнесение фрагментов по разным серверам (шардам) |

**Информация о местоположении фрагментов** хранится в **глобальном словаре данных** (также может быть распределённым).

**Важное требование:** СУБД должна обеспечивать механизм, при котором для программ и пользователей распределённая система воспринимается как единая централизованная БД.

---

#### Репликация

| Тип | Описание | Преимущества | Недостатки |
|---|---|---|---|
| **Полная репликация** | Синхронизируемые копии одной БД на всех компьютерах | Максимальная безопасность и доступность. Скорость выборки максимальна. Система работает, пока работает хотя бы один компьютер | Трудность синхронизации при обновлении. Между обновлениями копии могут отличаться |
| **Частичная репликация** | Копируются только отдельные фрагменты | Компромисс между доступностью и сложностью синхронизации | — |
| **Отсутствие репликации** | Каждый фрагмент имеет только одну копию | — | Нет избыточности, отказ компьютера → потеря данных |

**Eventual consistency (согласованность в конечном счёте):** в географически распределённых системах физически невозможно иметь полностью согласованную БД в каждый момент времени — требуется конечное время на передачу данных и синхронизацию. Некоторое время реплики могут быть рассогласованы, и это считается приемлемым.

---

### Виды репликации серверов

| Тип | Описание |
|---|---|
| **Multi-master (master-master)** | Несколько серверов могут принимать изменения |
| **Master-slave** | Один главный сервер (запись), остальные — подчинённые (чтение) |
| **Multi-source** | Несколько источников изменений |
| **Multi-server** | Несколько серверов |

**Проблемы репликации:**
- Репликация решает только проблемы чтения, а не записи.
- При частых изменениях данных синхронизация становится проблемой.
- Полностью автоматизировать синхронизацию невозможно — всегда требуются ручные решения конфликтов.

---

### Классификация распределённых СУБД

**Первый фактор — степень однородности:**

| Тип | Описание |
|---|---|
| **Однородная (гомогенная) РСУБД** | Все пользователи используют одно и то же программное обеспечение (СУБД). Данные на всех компьютерах контролируются той же СУБД |
| **Неоднородная (гетерогенная) РСУБД** | Используются разные СУБД, разные модели данных, типы, ограничения, языки |

**Второй фактор — степень автономности:**

| Тип | Описание |
|---|---|
| **Распределённая СУБД без локальной автономности** | Единая концептуальная схема, единый центр обработки запросов и транзакций. Части единой БД распределены по компьютерам |
| **Федеративная СУБД (FDBMS)** | Составлена из полностью автономных СУБД. Может включать СУБД с разными моделями данных. Поддерживает глобальную схему для распределённых запросов |

**Проблемы федеративных систем:**
- Отличия в поддерживаемых моделях данных.
- Отличия в именах таблиц и полей.
- Поля с одинаковыми именами могут обозначать разные данные.

---

### Достоинства и недостатки распределённых БД

| Достоинства | Недостатки |
|---|---|
| 1. **Разделяемость и локальная автономность** — географическая распределённость организации может быть отображена в распределении данных, локальный контроль | 1. **Повышение сложности** → увеличение стоимости и срока разработки |
| 2. **Управление на разных уровнях «прозрачности»** — реальное расположение данных должно быть скрыто от пользователя | 2. **Усложнение контроля целостности данных** |
| 3. **Увеличение стабильности и надёжности** — при выходе из строя отдельных частей система продолжает функционировать | 3. **Невозможность выполнения требований ACID** в ряде случаев → приходится жертвовать согласованностью в пользу **согласованности в конечном счёте (eventual consistency)** |
| 4. **Увеличение производительности** — фрагменты можно разместить там, где они наиболее нужны → снижение нагрузки на сеть | 4. **Усложнение контроля безопасности и секретности** — нужно защищать не только БД, но и сетевые соединения |
| 5. **Масштабируемость** — способность обрабатывать огромные объёмы данных за счёт наращивания ресурсов | |
| 6. **Увеличение гибкости за счёт модульности системы** | |

## 7.4. Многопользовательские базы данных

### 7.4.1. Блокировки (locks)

**Назначение:** организация многопользовательского доступа. На время выполнения операции доступ к объекту со стороны других пользователей временно запрещается или ограничивается.

**Блокировки относятся к пессимистичному подходу управления транзакциями.**

---

#### Виды блокировок (в порядке убывания строгости)

| Вид блокировки | Что разрешает | Что запрещает | Когда применяется |
|---|---|---|---|
| **1. Полная (исключающая, монопольная)** | Ничего | Любые операции | При изменении структуры объектов (на весь объект) |
| **2. Блокировка от записи (разделяемая)** | Чтение | Изменение данных, изменение структуры | На отдельные записи или страницы записей |
| **3. Предохраняющая блокировка от записи** | Защищает объект от полной блокировки или блокировки от записи со стороны других. Позволяет первому «захватившему» завершить модификацию | — | При совместном изменении данных несколькими пользователями |
| **4. Предохраняющая полная блокировка** | Защищает объект от полной блокировки со стороны других. Максимальный уровень совместного использования | — | Для обеспечения одновременного просмотра таблицы несколькими пользователями (никто не сможет изменить её структуру) |

---

#### Правила совмещения блокировок

| Блокировка A / Блокировка B | Полная | От записи | Предохр. от записи | Предохр. полная |
|---|---|---|---|---|
| **Полная** | Несовместима | Несовместима | Несовместима | Несовместима |
| **От записи** | Несовместима | Совместима | Несовместима | Совместима |
| **Предохраняющая от записи** | Несовместима | Несовместима | Совместима | Совместима |
| **Предохраняющая полная** | Несовместима | Совместима | Совместима | Совместима |

**Кратко:**
- Полная блокировка **несовместима** с любой другой.
- Блокировка от записи совместима с аналогичной и предохраняющей полной.
- Предохраняющая от записи совместима с обоими видами предохраняющих.
- Предохраняющая полная совместима со всеми, кроме полной.

---

#### Проблемы блокировок

| Проблема | Описание | Решение |
|---|---|---|
| **Бесконечное ожидание** | Транзакция, поступившая раньше, всё время отодвигается в конец очереди | Соблюдение очерёдности обслуживания |
| **Тупик (deadlock)** | Взаимные блокировки транзакций на пересекающихся множествах данных. Каждая ждёт освобождения ресурса, занятого другой | Введение **таймаута** (максимальной длительности ожидания). При превышении — прекращение одной из транзакций и её перезапуск через случайный промежуток |

---

### 7.4.2. Транзакции

**Определение:** транзакция — упорядоченная последовательность операторов обработки данных, которая переводит БД из одного согласованного состояния в другое.

**Связь с целостностью:** каждая транзакция начинается при целостном состоянии БД и должна оставить это состояние целостным после своего завершения. При нарушении — откат (ROLLBACK).

---

#### Виды ограничений целостности по времени проверки

| Тип | Когда проверяется | Что происходит при нарушении |
|---|---|---|
| **Немедленно проверяемые** | На уровне отдельных операторов | Отвергается только соответствующий оператор (отката транзакции нет) |
| **Откладываемые** | В конце транзакции (по умолчанию) | COMMIT заменяется на ROLLBACK. Можно принудительно проверить внутри транзакции специальным оператором |

---

#### Запуск транзакций

| Способ | Описание |
|---|---|
| **Явный** | С помощью оператора `BEGIN TRANSACTION` |
| **Неявный (AUTOCOMMIT)** | Для одной инструкции SQL. После выполнения автоматически выполняется `COMMIT` (откатить невозможно) |

---

#### Точки сохранения (savepoint)

**Назначение:** запоминание промежуточной копии состояния БД для возможности частичного отката.

**Сегмент отката (rollback segment, RBS)** — специальная область памяти на диске, в которую записывается информация обо всех незавершённых изменениях (старые значения атрибутов, часто и старые, и новые). Данные хранятся до завершения транзакции, затем могут быть перезаписаны.

---

#### Завершение транзакции (согласно стандарту ANSI/ISO)

| Событие | Действие |
|---|---|
| Выполнение `COMMIT` | Фиксация изменений |
| Выполнение `ROLLBACK` | Откат |
| Успешное завершение программы | Автоматический `COMMIT` |
| Аварийное завершение программы | Автоматический `ROLLBACK` |

Это позволяет организовать **оптимистический подход** управления транзакциями: достаточно организовать «опасные» операции обновления в рамках транзакций и не проводить многочисленных предварительных проверок.

---

### 7.4.3. Уровни изолированности транзакций

**Базовый принцип:** обновления, выполняемые транзакцией T1, не должны быть доступны для любой другой транзакции T2 до тех пор, пока не будет завершено выполнение T1.

**Уровни изолированности (в порядке возрастания строгости):**

```
READ UNCOMMITTED  <  READ COMMITTED  <  REPEATABLE READ  <  SERIALIZABLE
```

---

#### Четыре случая нарушения способности к упорядочению

| Нарушение | Описание | Пример |
|---|---|---|
| **1. Потерянные изменения** | Результаты успешно завершённой операции обновления одной транзакции перекрываются результатами другой | T1 изменяет A, до завершения T1 транзакция T2 также изменяет A → результат T1 потерян |
| **2. Неаккуратное считывание (чтение «грязных данных»)** | Транзакция читает данные, которые позже откатываются | T1 вставляет строку, T2 читает её, затем T1 откатывается → у T2 «фантомная» строка |
| **3. Неповторяемое считывание** | Транзакция повторно читает объект и видит его изменённым | T1 читает A, T2 изменяет A и завершается, T1 повторно читает A → видит новые данные |
| **4. Наличие фиктивных элементов (фантомов)** | Особый тип неповторяемого чтения при обновлении и чтении того же набора данных | Транзакция сначала обновляет набор данных, затем читает его, но чтение начинается раньше завершения обновления |

---

#### Сводная таблица (Таблица 7)

| Уровень изоляции | Потерянные изменения | Неаккуратное считывание | Неповторяемое считывание | Фиктивные элементы (фантомы) |
|---|---|---|---|---|
| **READ UNCOMMITTED** | Нет | **Да** | **Да** | **Да** |
| **READ COMMITTED** | Нет | Нет | **Да** | **Да** |
| **REPEATABLE READ** | Нет | Нет | Нет | **Да** |
| **SERIALIZABLE** | Нет | Нет | Нет | Нет |

**Минимальное требование к многопользовательской СУБД:** отсутствие потерянных изменений.

**Максимальное требование (часто):** отсутствие неповторяющихся чтений (хотя это ещё не гарантирует полной изолированности).

**Полная изолированность:** только SERIALIZABLE.

---

#### Сериализация транзакций

**Сериализация** — определение последовательности выполнения транзакций, при которой результат совместного выполнения эквивалентен результату последовательного выполнения этих же транзакций.

**Тривиальное решение:** действительно последовательное выполнение (но теряется параллельность).

**Более эффективное решение:** выполнение операторов разных транзакций в другом порядке с сохранением сериальности (например, только читающие транзакции или неконфликтующие по доступу к объектам).

---

### 7.4.4. Методы сериализации транзакций

#### Два базовых подхода

| Подход | Принцип |
|---|---|
| **На основе синхронизационных захватов (блокировок)** | Управление блокировками объектов БД |
| **На основе временных отметок (timestamp)** | Использование временных меток для определения порядка |

---

#### Метод двухфазного протокола блокировок

**Принцип:** перед выполнением операции над объектом r транзакция T запрашивает синхронизационный захват объекта r в режиме, зависящем от вида операции (захват по чтению или захват по записи).

**Проблема:** возможность возникновения **тупиков (deadlocks)**.

**Пример тупика:**
```
T1: монопольный захват p1, затем требуется совместный захват p2
T2: монопольный захват p2, затем требуется совместный захват p1
→ Ни одна не может продолжаться
```

**Устранение тупиков:**
1. Выбор «транзакции-жертвы» (на основе стоимости — время выполнения, число захватов, приоритет).
2. Откат транзакции-жертвы (полный или частичный).
3. Освобождение блокировок → продолжение других транзакций.

**Недостаток:** насильственное устранение тупиков — нарушение принципа изолированности пользователей.

---

#### Метод временных отметок (timestamp)

**Относится к оптимистическим алгоритмам.**

**Основная идея:** если транзакция T1 началась раньше транзакции T2, то система обеспечивает режим, как если бы T1 была целиком выполнена до начала T2.

**Алгоритм:**
1. Каждой транзакции T присваивается временная отметка t (время начала).
2. При выполнении операции над объектом r транзакция T помечает его своей временной отметкой и типом операции.
3. Перед выполнением операции транзакция T1 проверяет, не закончилась ли транзакция T, пометившая объект.
   - Если T закончилась → T1 помечает r и выполняет операцию.
   - Если T не закончилась → проверка конфликтности:
     - Если операции не конфликтны → T1 выполняет операцию.
     - Если конфликтны и t(T) > t(T1) (T «моложе») → откат T, T1 продолжает.
     - Если конфликтны и t(T) < t(T1) (T «старше») → T1 получает новую отметку и начинается заново.

**Недостатки метода временных отметок:**
- Потенциально более частые откаты транзакций (более грубое определение конфликтности).
- В распределённых системах сложно вырабатывать глобальные временные отметки.

**Достоинства:** не нужно распознавать и устранять тупики (что в распределённых системах очень дорого).

---

#### Алгоритм многовариантности (для запросов только на чтение)

**Принцип:** позволяет выполнить повторяемость считывания без наложения блокировок.

**Механизм:**
1. Каждый блок данных помечается маркером последней транзакции, модифицировавшей эти данные.
2. Если блок данных не модифицировался с момента начала чтения → данные считываются из этого блока.
3. Если данные были модифицированы → СУБД обращается к сегменту отката и считывает значения, относящиеся к моменту начала чтения.

## Ответы на контрольные вопросы

### Вопрос 1. Дайте характеристику технологий доступа к данным Microsoft.

**Эволюция технологий:**
- **Clipboard** — копирование и вставка через буфер обмена.
- **DDE** — протокол динамического обмена данными (медленный, ненадёжный).
- **OLE** — встраивание/связывание объектов, создание составных приложений.
- **COM** — двоичный стандарт взаимодействия объектов (не зависит от языка).
- **DCOM** — COM для распределённых систем.
- **ODBC** — универсальный доступ к SQL-БД через драйверы (абстрагирование от диалекта SQL).
- **OLE DB** — доступ к любым источникам данных (реляционные, объектные, документы, почта) через провайдеров.
- **DAO** — высокоуровневая объектная модель для Access (прямой доступ к ядру Jet).
- **ADO / ADO.NET** — высокоуровневые объектные модели для доступа к OLE DB-провайдерам, поддержка Internet/Intranet и .NET.

### Вопрос 2. Что такое распределённая БД?

**Распределённая БД (DDB)** — совокупность логически взаимосвязанных баз данных, распределённых в компьютерной сети. Для пользователя она должна выглядеть как единая централизованная БД.

### Вопрос 3. Что подлежит распределению в распределённой СУБД?

- **Данные** (фрагментация и репликация).
- **Функции СУБД** (обработка запросов, управление транзакциями, поддержка целостности).
- **Словарь данных** (глобальный словарь фрагментации, репликации и распределения).

### Вопрос 4. В чём состоит отличие архитектуры файл-сервер от архитектуры клиент-сервер?

| Аспект | Файл-сервер | Клиент-сервер |
|---|---|---|
| Передача данных | По сети передаются **файлы БД целиком** | По сети передаётся только **результат запроса** |
| Обработка данных | На компьютере клиента | На сервере |
| Загрузка сети | Высокая | Низкая |
| Многопользовательский доступ | Слабая поддержка | Хорошая поддержка |
| Целостность | Забота на клиенте | Обеспечивается сервером |

### Вопрос 5. Чем отличается удалённое представление от удалённого доступа к данным в архитектуре клиент-сервер?

| Архитектура | Удалённое представление (DBS) | Удалённый доступ к данным (RDA) |
|---|---|---|
| Логика обработки | На сервере (хранимые процедуры, триггеры) | На клиенте |
| Представление | На клиенте | На клиенте |
| Клиент | «Тонкий» | «Толстый» |
| Передача данных | Только результат запроса | Весь объём данных |
| Производительность | Высокая | Низкая при больших объёмах |
| Модификация | Труднее (ограничения хранимых процедур) | Легче |

### Вопрос 6. Назовите преимущества и недостатки распределённых БД.

| Преимущества | Недостатки |
|---|---|
| Разделяемость и локальная автономность | Повышение сложности, стоимости, срока разработки |
| Прозрачность расположения данных | Усложнение контроля целостности |
| Увеличение стабильности и надёжности | Невозможность выполнения ACID в ряде случаев (eventual consistency) |
| Увеличение производительности | Усложнение контроля безопасности (защита сети) |
| Масштабируемость | — |
| Гибкость за счёт модульности | — |

### Вопрос 7. Назовите основные способы организации распределённой БД с распределённым хранением данных.

1. **Фрагментация** (горизонтальная, вертикальная) + секционирование / шардинг.
2. **Репликация** (полная, частичная, отсутствие репликации).

### Вопрос 8. Что такое федеративная распределённая СУБД?

**Федеративная СУБД (FDBMS)** — распределённая СУБД, составленная из полностью автономных СУБД. Может включать СУБД, поддерживающие различные модели данных, типы, ограничения и языки манипулирования данными. Поддерживает глобальную схему для построения распределённых запросов.

### Вопрос 9. Перечислите способы решения проблем совместного доступа к общим данным в многопользовательских БД.

1. **Блокировки** (полная, от записи, предохраняющие).
2. **Транзакции** (с возможностью отката и фиксации).
3. **Уровни изолированности транзакций** (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE).
4. **Методы сериализации транзакций** (двухфазный протокол блокировок, метод временных отметок, алгоритм многовариантности).

### Вопрос 10. Перечислите виды блокировок и правила их совмещения.

**Виды блокировок (в порядке убывания строгости):**
1. Полная (исключающая, монопольная)
2. Блокировка от записи (разделяемая)
3. Предохраняющая блокировка от записи
4. Предохраняющая полная блокировка

**Правила совмещения (см. таблицу выше):**
- Полная несовместима с любой другой.
- От записи совместима с аналогичной и предохраняющей полной.
- Предохраняющая от записи совместима с обоими видами предохраняющих.
- Предохраняющая полная совместима со всеми, кроме полной.

### Вопрос 11. Какой из уровней изоляции транзакций обеспечивает максимальную изолированность пользователей?

**SERIALIZABLE.** При этом уровне гарантируется отсутствие потерянных изменений, неаккуратного считывания, неповторяемого считывания и фиктивных элементов.

### Вопрос 12. Какие вы знаете случаи нарушения способности к упорядочению транзакций?

1. Потерянные изменения
2. Неаккуратное считывание (чтение «грязных данных»)
3. Неповторяемое считывание
4. Наличие фиктивных элементов (фантомов)

### Вопрос 13. На каком уровне изоляции транзакций может происходить чтение «грязных данных»?

**READ UNCOMMITTED.** На этом уровне допускается неаккуратное считывание.
